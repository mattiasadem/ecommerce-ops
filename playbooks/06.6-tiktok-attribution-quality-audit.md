# Move #6.6 — TikTok Pixel + Events API (EAPI) quality audit

> **Why this move fills the third-platform gap that Move #6.5 missed.** Move #6.5 audits Meta CAPI + Google Enhanced Conversions + GA4 ↔ Triple Whale — the **two** major ad-platform attribution stacks. But TikTok is now the third-largest paid acquisition channel for most DTC stores (2025 share: 18–25% of paid social spend, second only to Meta at 35–45%). The audit gap is silent: operators don't notice they're missing TikTok attribution quality because Triple Whale's "TikTok Ads" integration shows a green checkmark as long as the events API token is present — it doesn't check whether the events are matching, whether the dedup is real, or whether the pixel is firing on every page. Move #6.5's "if Meta CAPI match rate is healthy, the rest is fine" intuition is wrong on stores with >$5k/mo TikTok spend. A TikTok pixel that drops to 40% coverage (because the pixel snippet was lost in a theme update) silently degrades TikTok's learning algorithm, costs $2k–$5k/mo in wasted spend on a mid-tier store, and is invisible until the operator manually checks the TikTok Events Manager diagnostics. **This audit catches the gap before it costs you a quarter of bad TikTok spend decisions.**
>
> **Honest read.** Move #6.6 follows the exact same pattern as Move #6.5 but scoped to TikTok's 4 diagnostic surfaces (vs Move #6.5's 6). The thresholds are TikTok-specific: TikTok's own documentation states that **event_id match rates below 70%** silently degrade Smart+ campaign optimization (vs Meta's 80% threshold); below **50%** TikTok throttles EAPI delivery entirely (vs Meta's 70%). TikTok's dedup ratio band is wider because TikTok pixel is newer and over-fires more often during initial install — `[0.7, 1.6]` is the canonical band vs Meta's `[0.8, 1.5]`. The cohort-sync audit is dropped because TikTok's cohort-attach to Klaviyo is rare (most operators use TikTok for top-of-funnel acquisition, not for re-engagement). The drift audit is dropped because TikTok's diagnostic export cadence is weekly at best, not daily, so week-over-week drift detection is noisy. **4 gates vs Move #6.5's 6 — by design, not by accident.**
>
> **Companion script:** `scripts/tiktok_attribution_audit.py` (new this tick, ~520 lines) is a hermetic measurement-quality audit that validates 4 numeric-quality gates against JSON fixtures the operator exports from TikTok Events Manager. It does NOT require API credentials and cannot authenticate against TikTok — instead, it reads the local diagnostic-export JSONs (TikTok Events Manager → Diagnostics tab) and runs the same 4 quality gates the playbook documents. Default bootstrap generates canonical-pass fixtures so the operator can run the audit immediately and see all-green.

## Goal

In a single 1.5–2 hour session, stand up a weekly TikTok attribution-quality audit cadence that:

1. **Confirms TikTok EAPI event_id match rate is ≥ 85%.** TikTok's docs say match rates below 70% silently degrade Smart+ campaign optimization; below 50% TikTok throttles EAPI delivery entirely. The threshold is published in `MIN_TIKTOK_EAPI_MATCH_RATE_PCT = 85.0` in the script and pinned by a TDD test. The threshold is **less strict** than Meta's 90% because TikTok's install is newer and a 85% match rate is realistically the best most stores achieve in steady state.
2. **Confirms TikTok EAPI dedup ratio is in [0.7, 1.6].** `pixel_events / eapi_events` should be 0.7–1.6 — outside that band means one side is over-firing (double-counting risk) or under-firing (missing attribution). The wider band vs Meta (0.8–1.5) reflects TikTok's pixel-over-firing pattern during initial install.
3. **Confirms TikTok Pixel coverage is ≥ 90% of expected pageviews.** Below 70% means the pixel is not on every page (usually a theme.liquid edit was missed during a theme update); 70–90% is normal ad-blocker-induced loss; 90%+ is the canonical-pass band.
4. **Confirms TikTok hashed-email (Advanced Matching) coverage is ≥ 75%.** TikTok's Advanced Matching is the analogue to Meta's CAPI + Google's Enhanced Conversions — without it TikTok can only match on coarse signals (device ID, IP) and attribution falls back to last-click. 75% is the realistic canonical-pass band (vs Meta's 90% + Google's 80%).

**Out of scope:** Snapchat Pixel (CAPI) — too niche for 2025 DTC volumes (<3% of paid social share); Pinterest Tag (CAPI) — same; LinkedIn Insight Tag (CAPI) — B2B-only; programmatic (DV360, TTD) — separate stack entirely; in-house attribution engineering.

## Which audit cadence fits your team

Pick one of three cadences. The decision is driven by **TikTok paid-spend velocity + how often you make TikTok-dependent decisions**, not by how thorough you want to be.

| Path | Team profile | Audit cadence | Tool | Why this one |
|---|---|---|---|---|
| **A** | Solo operator, <$2k/mo TikTok spend, ships TikTok decisions monthly | **Monthly** audit, run on the 1st of each month | `scripts/tiktok_attribution_audit.py` (manual invocation) | Cheapest cadence. Monthly catches drift between quarters; the decision cycle is monthly so more frequent audits are noise |
| **B** | Small team (2–5 people), $2k–$20k/mo TikTok spend, ships TikTok decisions weekly | **Weekly** audit, run Monday morning | Same script, plus a cron job that emails the JSON output | **The default for Move #6.6 shops.** Weekly catches the common EAPI match-rate regressions within 1 decision cycle; emails the JSON to a Slack channel for visibility |
| **C** | Larger team (5+ people), $20k+/mo TikTok spend, ships TikTok decisions daily | **Daily** audit, run at 07:00 UTC | Same script, plus a Vercel cron or AWS EventBridge that calls the script and pushes the JSON to a dashboard | Above $20k/mo the cost of a single bad TikTok decision is meaningful (one ad-set with bad attribution can burn $300/day). Daily catches issues within 24h |

**Default for this playbook: Path B (weekly audit, Monday morning).** Path A for solo operators who don't need weekly visibility. Path C only for stores spending $20k+/mo TikTok where 24h regression detection matters.

**Decision rule of thumb:** if your TikTok spend is ≥$2k/mo AND you have ≥2 TikTok-dependent decisions per month (reallocate budget between Spark Ads and In-Feed, launch a new Spark Ad from a winning organic, kill an underperformer), Path B is the minimum. Below $2k/mo TikTok spend the decision cycle is too slow for weekly audits to add value.

## Step-by-step

Five steps. Steps 1–3 are the one-time setup (~30 min). Steps 4–5 are the recurring weekly cadence (~30 min per audit). The whole playbook is shippable in 1.5–2 hours of initial setup + ongoing 30 min/wk.

### Step 1 — Export the 4 diagnostic JSON files (one-time, ~15 min)

For each TikTok diagnostic surface, navigate to the diagnostic export, copy the values into the JSON fixture template, and save the file to `./tiktok_fixtures/`. The script's `--bootstrap` flag generates a starter fixture set in the right directory.

**TikTok Events Manager diagnostic (last 7d):**

1. Go to ads.tiktok.com → Assets → Events → click your pixel → "Diagnostics" tab → "Event Quality" subtab.
2. Set the time range to "Last 7 days."
3. Note four numbers: `Pixel events` (count of `Purchase` events fired by the browser pixel), `Server events` (count of `Purchase` events fired by EAPI/Events API), `Matched events` (count of `Purchase` events matched by `event_id`), and `Expected orders` (count of `Order created` webhooks from Shopify in the same window — get this from `Shopify → Analytics → Total orders` filtered to last 7d).
4. Paste these into `tiktok_fixtures/tiktok_eapi_match.json`:
   ```json
   {
     "pixel_events": 1000,
     "eapi_events": 950,
     "matched_events": 870,
     "expected_orders_last_7d": 950
   }
   ```
5. Separately, for Gate C, get `PageView event count` from the same Diagnostics tab → "Event Quality" → filter to `PageView` events → note the count. Get `Total sessions` from `Shopify → Analytics → Total sessions` for the same 7d window. Paste into `tiktok_fixtures/tiktok_pixel_coverage.json`:
   ```json
   {
     "pixel_fired_count": 9000,
     "expected_pageviews": 10000
   }
   ```

**TikTok Advanced Matching diagnostic (last 7d):**

1. Go to ads.tiktok.com → Assets → Events → click your pixel → "Advanced Matching" subtab.
2. Note `Matched events with hashed identifier` (the count of `Purchase` events that included at least one user identifier — email, phone, or external_id — that TikTok was able to hash and match) and `Total events` (the count of all `Purchase` events in the window).
3. Compute `hashed_identifier_coverage_pct = matched_with_identifier / total_events * 100` and paste both into `tiktok_fixtures/tiktok_advanced_matching.json`:
   ```json
   {
     "hashed_identifier_coverage_pct": 80.0,
     "matched_with_identifier": 760,
     "total_events": 950
   }
   ```

### Step 2 — Run the audit (one-time, ~5 min)

From the workspace root:

```bash
python3 scripts/tiktok_attribution_audit.py
```

Expected output on the bootstrapped canonical-pass fixtures:

```
======================================================================
TikTok Attribution Quality Audit — Move #6.6 Verification
======================================================================
Fixture dir: ./tiktok_fixtures/

  [✓] tiktok_eapi_match_rate
      ✓ Match rate: 91.6% (target ≥85.0%) | ...
  [✓] tiktok_pixel_coverage
      ✓ Pixel coverage: 90.0% (target ≥90.0%) | ...
  [✓] tiktok_advanced_matching
      ✓ Hashed identifier coverage: 80.0% (target ≥75.0%) | ...

Summary: 3/3 gates passed

ALL GATES PASSED — TikTok attribution stack is producing actionable signal.
======================================================================
```

(3 gates displayed because Gate B — dedup ratio — is a sub-check inside Gate A and does not get its own gate entry; this matches the Move #6.5 pattern.)

Exit code 0 = all gates pass. Exit code 1 = at least one gate fails. Per-gate run via `python3 scripts/tiktok_attribution_audit.py --check tiktok_eapi_match_rate`. JSON output via `--json` (for cron / CI integration).

### Step 3 — Wire the cadence (one-time, ~5 min)

Pick one of three cadences (Path A / B / C from the table above) and wire it.

**Path B (default — weekly Monday morning audit):**

```bash
# Add to crontab (crontab -e):
# Weekly Monday 09:00 — run audit, save JSON, email if any gate fails
0 9 * * 1 cd /path/to/ecommerce-ops && python3 scripts/tiktok_attribution_audit.py --json > /tmp/tiktok_audit_$(date +\%Y\%m\%d).json 2>&1; if [ $? -ne 0 ]; then cat /tmp/tiktok_audit_*.json | mail -s "TikTok Attribution Audit FAILED" you@example.com; fi
```

**Path A (monthly) and Path C (daily)** — same shape, different cron cadence.

### Step 4 — Investigate any gate failures (recurring, ~15 min per failure)

When a gate fails, the operator should:

1. Read the `detail` line of the failed gate — it names the exact threshold + actual value + the diagnostic export to re-pull.
2. Read the `remediation` line — it names the specific fix for the failure mode.
3. Re-export the diagnostic (in case the previous export was stale) and re-run the audit.
4. If still failing, walk the remediation steps in order. The most common fixes are documented in the `## Common pitfalls` section.

The full remediation table is in the `## Verification` section below.

### Step 5 — Document cadence + on-call rotation (one-time, ~10 min)

Write down:

1. **The weekly audit owner** (one specific person, not "the marketing team").
2. **The Slack channel** where the JSON output is posted (e.g. `#tiktok-attribution`).
3. **The on-call rotation** for when a gate fails (one primary + one backup, with phone numbers).
4. **The escalation path** — when does a failed gate become a P0 vs P1? Default rule: any gate failure lasting >24h is P0 because it indicates an active attribution regression that is silently costing money.

Save this to Notion or your team's runbook. The audit is a process, not a one-time task.

| 5 | **Forgetting to re-export after a Shopify webhook delay.** The audit uses last-7d exports but the Shopify order webhook was delayed by 6 hours, so `expected_orders_last_7d` undercounts by ~3%. | Gate A fails with `coverage 91.8% < 95.0% target` despite no actual coverage drop. | Re-export after waiting 6h for the webhook to catch up. | 6 | **Measuring dedup ratio when EAPI is offline.** TikTok EAPI token expired and no events are being sent server-side; `eapi_events = 0` triggers the dedup guard's "CAPI events = 0" failure. | Gate A fails with `CAPI events = 0 (TikTok cannot optimize without server-side events)`. | Re-authenticate the EAPI token in TikTok Events Manager → Settings → Web Events API → regenerate token → paste into the integration. |
| 7 | **Comparing against an outdated baseline.** The audit's match rate is 91.6% but the operator remembers last week's was 95%; assumes the audit regressed and tightens the threshold. | Operator edits the threshold from 85% to 92% thinking they're "improving" the audit. | The audit's match rate is a **measurement**, not a goal. | 9 | **Running the audit on a partial week.** The audit's measurement window is `last 7d` but the operator only exported 3 days of data because they're "trying it out." | All gates pass with artificially high numbers (small sample = low noise = high match rate). | Wait 7 full days before running the first audit. ## Verification

Four gates, runnable end-to-end. Each gate has a `Measurement` (what to compute), a `Target` (the threshold), and a `Pass criterion` (how to know it passed).

### Gate A — TikTok EAPI event_id match rate + dedup ratio + coverage

**Measurement:** `matched_events / expected_orders_last_7d * 100` for match rate; `pixel_events / eapi_events` for dedup ratio; `matched_events / expected_orders_last_7d * 100` for coverage.

**Target:** match rate ≥ 85%, dedup ratio in [0.7, 1.6], coverage ≥ 95% of expected orders.

**Pass criterion:** all three sub-checks pass simultaneously. Any single sub-check failure fails Gate A.

**How to verify the gate is testing the right thing:**
1. Edit `tiktok_eapi_match.json` to set `matched_events = 0`. Re-run the audit. Gate A should fail with `match rate 0.0% < 85.0% target`.
2. Restore `matched_events = 870`. Edit `eapi_events = 50` (way out of dedup band). Gate A should fail with `dedup ratio 20.00 outside [0.7, 1.6]`.
3. Restore `eapi_events = 950`. Edit `matched_events = 900, expected_orders_last_7d = 1000` (90% coverage). Gate A should fail with `coverage 90.0% < 95.0% target`.

### Gate C — TikTok Pixel coverage (browser-side intent signals)

**Measurement:** `pixel_fired_count / expected_pageviews * 100`.

**Target:** ≥ 90% (canonical-pass band). 70–90% is normal ad-blocker-induced loss; < 70% indicates the pixel is not on every page.

**Pass criterion:** coverage ≥ 90%.

**How to verify the gate is testing the right thing:**
1. Edit `tiktok_pixel_coverage.json` to set `pixel_fired_count = 5000`. Gate C should fail with `Pixel coverage 50.0% < 90.0% target`.
2. Restore `pixel_fired_count = 9000`. Gate C should pass.

### Gate D — TikTok Advanced Matching coverage

**Measurement:** `matched_with_identifier / total_events * 100`.

**Target:** ≥ 75%. < 60% indicates Advanced Matching is not enabled or the Klaviyo sync is not sending hashed identifiers.

**Pass criterion:** coverage ≥ 75%.

**How to verify the gate is testing the right thing:**
1. Edit `tiktok_advanced_matching.json` to set `hashed_identifier_coverage_pct = 50.0`. Gate D should fail with `hashed identifier coverage 50.0% < 75.0% target`.
2. Restore to 80.0. Gate D should pass.
