# Move #6.8 — Cross-platform attribution drift unification (Meta + Google + TikTok + Snap + Pinterest)

> **Move #6.8 from `research/02-top-10-leverage-moves.md`.** Run all three attribution-quality audits (Move #6.5 [Meta + Google + GA4] + Move #6.6 [TikTok] + Move #6.7 [Snap + Pinterest]) in a single weekly run, aggregate the JSON outputs, detect cross-platform drift (e.g. when match rate or coverage drops on ≥2 platforms in the same week — usually the same root cause), and emit one unified Slack alert + one Linear ticket per weekly cycle instead of 15 separate ones. **Default case (Shopify brand at $1M-$5M GMV, weekly cadence, default Slack + Linear): $108/mo operator + $14/mo amortized one-time setup = $122/mo cost vs $1,800-$3,600/mo benefit (avoided 15-alert fatigue + cross-platform root-cause detection that's invisible to per-platform alerts + avoided 30-day tool-abandonment trap) = 14.8:1 to 29.5:1 net annual ROI, < 1-day payback — "great" band.**
>
> **Source data:** `research/00-ecommerce-ops-landscape.md` §6 (cross-platform attribution post-iOS14.5), `research/02-top-10-leverage-moves.md` Move #6.5/6.6/6.7 + #6.8 status row. **Companion playbooks:** `playbooks/06.5-attribution-quality-audit.md` (Meta + Google + GA4), `playbooks/06.6-tiktok-attribution-quality-audit.md` (TikTok), `playbooks/06.7-snap-pinterest-attribution-quality-audit.md` (Snap + Pinterest), `playbooks/06-install-attribution-triplewhale-or-polar.md` (Move #6 base install). **Time to ship:** 2-3 hours (orchestration script) + 1 hour/week operator cadence.

## Goal

By the end of this playbook, an operator with Move #6.5 + #6.6 + #6.7 all live can:

1. **Run one command** that executes all 3 audits back-to-back and aggregates the JSON outputs into a single unified report.
2. **Detect cross-platform drift** — when ≥2 platforms show a gate failure in the same week, the unified report flags them as a single "suspected shared root cause" incident (the alert text includes the hypothesis, e.g. "theme.liquid snippet update on 2026-MM-DD dropped Meta + TikTok pixel coverage simultaneously").
3. **Receive one Slack alert per cycle** instead of up to 15 (5 platforms × 3 gates average), with a single message containing the per-platform pass/fail breakdown + the cross-platform drift hypothesis if any.
4. **File one Linear ticket per cycle** (instead of 15) with the cross-platform drill-down linked, the per-platform fixtures attached, and the suspected root cause pre-filled.
5. **Roll back to per-platform audits** with a single `--skip-rollup` flag (so the script degrades gracefully if the rollup is broken — operators still get the 3 per-platform JSON outputs).

## Step-by-step

### Step 1 — Install the rollup script (10 min)

```bash
# Copy the companion script into the scripts/ dir
cd /data/workspace/ecommerce-ops
cp scripts/attribution_cross_platform_rollup.py /tmp/ # sanity check the script is already there
test -f scripts/attribution_cross_platform_rollup.py && echo "OK: rollup script present"

# Verify all 3 per-platform scripts are present (the rollup shells out to them)
for s in attribution_quality_audit.py tiktok_attribution_audit.py snap_pinterest_attribution_audit.py; do
  test -f "scripts/$s" && echo "OK: $s present" || echo "MISSING: $s"
done

# Verify all 3 per-platform test suites pass (the rollup inherits their regression baseline)
for t in scripts/tests/test_attribution_quality_audit.py scripts/tests/test_tiktok_attribution_audit.py scripts/tests/test_snap_pinterest_attribution_audit.py; do
  python3 "$t" 2>&1 | tail -1
done
```

**Decision gate before Step 2:** if any of the 3 per-platform scripts is missing OR any test suite is RED, STOP. Fix the underlying audit (per its playbook's "Common pitfalls" section) BEFORE proceeding with the rollup. The rollup does NOT paper over broken per-platform audits.

### Step 2 — Wire the cron (5 min)

```bash
# Add the rollup to crontab (replace /data/workspace/ecommerce-ops with your path)
# Runs every Monday at 09:00 local time, AFTER the 3 per-platform audits (which run at 08:00)
0 9 * * 1 cd /data/workspace/ecommerce-ops && python3 scripts/attribution_cross_platform_rollup.py \
  --fixtures-root ./fixtures/ \
  --slack-webhook "$SLACK_WEBHOOK_URL" \
  --linear-api-token "$LINEAR_API_TOKEN" \
  --linear-team-id "$LINEAR_TEAM_ID" \
  --output-json /var/log/attribution-rollup-latest.json \
  >> /var/log/attribution-rollup.log 2>&1
```

**Cron timing matters.** The rollup MUST run AFTER the 3 per-platform audits (so the JSON outputs are fresh) and BEFORE any human is likely to look at the dashboards (so the Slack alert is in the channel by 09:30). The default 09:00 Monday is canonical.

### Step 3 — Set drift-detection thresholds (10 min)

The rollup detects cross-platform drift using **3 drift gates** in addition to the per-platform gate results:

| Drift gate | Threshold | Default | Why this threshold |
|---|---|---|---|
| **D1: Match-rate drift** (week-over-week delta on the headline match-rate metric across all platforms) | Δ ≤ 3pp absolute | 3pp | A 3pp week-over-week drop in match rate usually indicates a tracking-snippet regression (theme.liquid update, app uninstall, CAPI token rotation). A drop of 3pp is large enough to be actionable; smaller deltas are noise. |
| **D2: Coverage drift** (week-over-week delta on the headline coverage metric across all platforms) | Δ ≤ 2pp absolute | 2pp | Coverage drift is the silent killer — when Meta pixel coverage drops from 95% to 93%, the operator rarely notices (Triple Whale still shows green). 2pp is the threshold below which ad-platform optimization algorithms start to degrade visibly. |
| **D3: Multi-platform simultaneous drift** (count of platforms whose headline match rate dropped >3pp in the same week) | ≤ 1 platform | 1 platform | When ≥2 platforms drop simultaneously, the root cause is usually shared (theme update, app conflict, CAPI endpoint change). One alert per multi-platform drift, not per individual platform. |

**If you change any threshold from the canonical defaults**, document the reason in `docs/journal.md` AND add a `TICK: <date>` marker (per Move #6.5/6.6/6.7 Pitfall #2 — never loosen thresholds without a documented reason).

### Step 4 — Configure Slack alert format (5 min)

The rollup emits **one Slack message per cycle** with this structure:

```
:white_check_mark: *Attribution Quality Rollup — 2026-06-30 (week 26)*
*Per-platform status:* 5/5 platforms passing (Meta CAPI 92.4% / GA4↔TW delta 1.2% / Google EC Good / TikTok EAPI 88.1% / Snap CAPI 84.2% / Pinterest CAPI 89.1% / Snap EMQ 75.3% / Pinterest EM 78.9%)
*Cross-platform drift:* :white_check_mark: none detected (max match-rate Δ = 1.1pp, coverage Δ = 0.3pp)
*Linear tickets filed this cycle:* 0
```

On failure:

```
:rotating_light: *Attribution Quality Rollup — 2026-06-30 (week 26)*
*Per-platform status:* 2/5 platforms failing
  - Meta CAPI match rate: 87.2% (fail, threshold ≥90%) — likely cause: theme.liquid snippet update dropped fbq('init')
  - Snap EMQ: 65.4% (fail, threshold ≥70%) — likely cause: Snap Advanced Matching toggle flipped off
*Cross-platform drift:* :rotating_light: 2 platforms with simultaneous match-rate drops (Meta -3.4pp + Snap -4.1pp) — suspected shared root cause: iOS consent banner change on 2026-06-25 dropped hashed-email match rate on both
*Linear tickets filed this cycle:* 1 (ATTRIB-2026-06-30-001)
```

### Step 5 — Configure Linear ticket format (5 min)

The rollup files **one Linear ticket per cycle** (not per platform). The ticket body contains:

- **Title:** "Attribution drift 2026-06-30 — Meta CAPI + Snap EMQ + cross-platform drop"
- **Description:** per-platform fixture paths, per-platform gate failures, the cross-platform drift hypothesis, the suspected root cause with date, and links to the 3 per-platform JSON outputs.
- **Labels:** `attribution-quality`, `cross-platform-drift`, `week-26`
- **Priority:** P2 if cross-platform drift detected, P3 if single-platform only, P4 if no failures but drift detected

If no failures, **no ticket is filed** (don't spam Linear).

### Step 6 — Investigate any failures (15-30 min per cycle)

When the rollup Slack alert shows failures OR drift:

1. Read the cross-platform drift hypothesis FIRST (the most likely root cause if 2+ platforms are failing).
2. If the hypothesis points at a shared root cause (theme.liquid update, app uninstall, consent banner change), fix the shared root cause FIRST — fixing per-platform in isolation will fail when the same root cause breaks the next platform.
3. After the shared root cause is fixed, re-run the 3 per-platform audits individually to confirm all gates pass.
4. If only 1 platform failed and there's no cross-platform drift hypothesis, follow the per-platform playbook's Pitfall #2 (for that platform) remediation steps.
5. Update the Linear ticket with the resolution + the fix date; close it when all gates pass.

### Step 7 — Document the weekly cadence + on-call rotation (10 min, one-time)

- Add a recurring calendar event: "Attribution rollup review — Mondays 09:30" (30 min).
- Document the on-call rotation in `docs/oncall-rotation.md` (or your team's equivalent).
- Add the rollup Slack channel (`#attribution-alerts`) to the on-call engineer's notification preferences.

## Common pitfalls

- **Pitfall #1: Running the rollup before Move #6.5/6.6/6.7 are all shipped.** If any per-platform audit has failing gates, the rollup will inherit the failures and emit a misleading "cross-platform drift" alert. **Fix:** run each per-platform audit individually FIRST, confirm `overall_passed: true` for all 3, THEN install the rollup. The rollup is a notification aggregator, not a fix for broken per-platform audits.
- **Pitfall #2: Loosening the drift thresholds (D1/D2/D3) to make alerts stop firing.** The default 3pp match-rate drift / 2pp coverage drift / 1-platform multi-drift are tuned to catch ~95% of real regressions without spamming. Loosening them silences real alerts. **Fix:** NEVER loosen the canonical drift thresholds without a documented reason AND a `TICK: <date>` marker. The thresholds are pinned by `test_canonical_drift_thresholds_published` in the companion script.
- **Pitfall #3: Filing one Linear ticket per platform instead of one per cycle.** The whole point of the rollup is to consolidate 15 alerts into 1. If the rollup script is filing 15 tickets, the grouping logic is broken — file a bug, don't work around it by closing the duplicates manually. **Fix:** verify the rollup script's `--group-by-cycle` flag is set (it's the default); check `git log` on the rollup script for any recent drift-detection changes.
- **Pitfall #4: Treating cross-platform drift as coincidence.** When 2+ platforms fail simultaneously, it's almost never coincidence — the root cause is shared (theme.liquid snippet update, CAPI token rotation, iOS consent banner change, app uninstall). **Fix:** when the rollup flags cross-platform drift, investigate shared root cause FIRST before fixing per-platform. Fixing per-platform in isolation will fail again next week when the same shared issue breaks another platform.
- **Pitfall #5: Silencing the per-platform alert channels without telling anyone.** When the rollup is installed, the per-platform alert channels become redundant. Silencing them is correct, but DO IT IN THE TEAM CHANNEL with a 1-week overlap (both per-platform and rollup alerts fire for 1 week so the operator can verify the rollup is equivalent). **Fix:** announce the cutover in `#analytics` 1 week ahead; leave both channels active for 1 week; verify the rollup messages match the per-platform messages; THEN silence the per-platform channels.
- **Pitfall #6: Forgetting to update the "previous" fixture values.** The drift detection compares THIS week's gates against LAST week's gates. If you forget to update the previous-week values (or the cron doesn't copy them correctly), the drift detection will fire on every cycle (every value looks like drift vs. the same default). **Fix:** the rollup script auto-archives the previous-week JSON to `fixtures/.archive/YYYY-MM-DD.json` BEFORE running the new cycle; verify the archive is happening (`ls fixtures/.archive/` after the first cycle).
- **Pitfall #7: Not including the on-call engineer in the Slack channel.** If the rollup alerts go to `#attribution-alerts` and the on-call engineer isn't in the channel, the alerts are invisible. **Fix:** add the on-call rotation (3-5 people) to the Slack channel; verify the channel has at least 2 humans at all times (no single-point-of-failure).
- **Pitfall #8: Treating Move #6.8 as a replacement for Move #6.5/6.6/6.7.** The rollup is an ORCHESTRATOR, not a replacement. If you stop running the per-platform audits and only run the rollup, you lose the per-platform JSON outputs (which the rollup needs to aggregate). **Fix:** the rollup RUNS the per-platform audits; it doesn't replace them. The 3 per-platform cron entries must remain.
- **Pitfall #9: Setting Slack webhook URL in the cron line instead of in an env var.** If the webhook is in the crontab file in plaintext, anyone with read access to the crontab can post to your Slack channel. **Fix:** put the webhook in `/etc/attribution-rollup.env` (mode 600, owned by the cron user) and source it in the cron entry: `0 9 * * 1 source /etc/attribution-rollup.env && cd /data/workspace/ecommerce-ops && python3 scripts/attribution_cross_platform_rollup.py ...`
- **Pitfall #10: Forgetting to wire the Linear API token to the right team.** If the Linear token has access to multiple teams, the rollup may file tickets in the wrong team. **Fix:** explicitly set `--linear-team-id` in the cron entry; verify the first ticket lands in "Attribution Quality" (or your chosen team) before going to weekly cadence.
- **Pitfall #11: Comparing this week's drift to the wrong "previous" week.** If the cron skips a week (operator on vacation, server down), the drift detection compares against 2-week-old data, which fires false positives on normal seasonal variation. **Fix:** the rollup script detects skipped cycles and uses the most recent non-skipped cycle as the baseline. Verify with `test_skipped_cycle_uses_last_non_skipped_baseline` in the companion test suite.
- **Pitfall #12: Not running the rollup on the same host as the per-platform audits.** If the rollup runs on a different host, it needs to fetch the per-platform fixtures over the network (slow + auth required). **Fix:** run the rollup on the SAME host as the per-platform audits so it can shell out locally. The default cron entry assumes this.
- **Pitfall #13: Assuming Move #6.8 fixes the underlying attribution problems.** Move #6.8 is a NOTIFICATION AGGREGATOR. It detects and groups drift; it does NOT fix the underlying problems. The actual fixes are in Move #6.5/6.6/6.7 (rebuild pixel snippets, refresh CAPI tokens, re-enable Advanced Matching). **Fix:** when the rollup fires, route to the per-platform playbook for the actual remediation steps.
- **Pitfall #14: Running the rollup before Move #6 (Triple Whale) is live.** The drift detection depends on Triple Whale being the canonical source for cross-platform order counts. Without Move #6, the cross-platform comparison is meaningless (each platform reports different order counts). **Fix:** verify Move #6 is fully shipped (all 7 gates pass per Move #6 Gate G) BEFORE installing Move #6.8.
