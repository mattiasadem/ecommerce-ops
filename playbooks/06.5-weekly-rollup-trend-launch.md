# Move #6.5 weekly-rollup-trend launch — multi-cycle trend analysis on top of Move #6.8 cross-platform attribution rollup

## Goal

Close the canonical slow-erosion blind spot that Move #6.8 cross-platform attribution drift unification cannot see: 1-vs-1 drift detection compares only the LAST cycle vs the cycle before that, so a match-rate losing 0.4 pp per week for 6 weeks (= −2.4 pp cumulative) stays below the canonical 3.0 pp 1-vs-1 drift threshold each individual week while the trend is clearly downward. Move #6.5 weekly-rollup-trend consumes the last N weeks of Move #6.8 rollup JSON outputs, computes the per-platform match-rate + coverage trend direction (improving / declining / stable), and fires a Slack-compatible alert when cumulative drift OR consecutive weekly decline crosses the canonical thresholds.

The trend script is a Track 9-12 hardening follow-up to Move #6.8 — it sits ATOP the existing Move #6 attribution-audit substrate (Move #6.5 + #6.6 + #6.7 + #6.8 + #6.9 + #6.10 all shipped 2026-06-25 to 2026-07-10) as the canonical multi-cycle trend-comparison companion to Move #6.10's per-cycle alert webhook.

## Which trend cadence fits your team

Pick the cadence that matches your operational capacity. The trend script is hermetic — it runs against `--rollup-dir` of past Move #6.8 rollup JSON files; it does NOT need a live wire.

- **A. Solo operator / <$5k/mo paid spend** — monthly trend run (the Move #6.8 weekly cadence is overkill for solo; the trend script catches the slow-erosion blind spot even at monthly cadence because the cumulative window is 12 weeks = 3 monthly cycles).
- **B. Small team / $5k–$50k/mo paid spend — DEFAULT** — weekly trend run alongside the Move #6.8 weekly rollup, so the operator gets BOTH the 1-vs-1 per-cycle drift alert (Move #6.8 + Move #6.10) AND the 12-week cumulative trend (Move #6.5 weekly-rollup-trend) on every cycle.
- **C. Larger team / $50k+/mo paid spend** — daily trend run with a tighter window (override `--trend-window-weeks` to 4 — only meaningful if you have ≥4 weeks of daily Move #6.8 rollup data; if your Move #6.8 cadence is weekly, drop this path and stay on Path B weekly).

Decision rule: if you already run Move #6.8 weekly (the canonical default), run the trend script on the SAME weekly cadence (Path B). The trend script is cheap — it just re-reads past Move #6.8 JSONs and computes sums — so the marginal cost of weekly cadence is <1 minute of cron time.

   - **advanced_matching_toggle** — did any platform's Advanced Matching toggle flip? Check `Meta Events Manager > Data Sources > Pixel > Settings > Advanced Matching` + `TikTok Events Manager > Pixel > Advanced Matching` + `Snap Pixel Manager > Settings > Advanced Matching` + `Pinterest Tag Manager > Enhanced Match`.
4. **Run `scripts/attribution_cross_platform_rollup.py` with `--dry-run` + `--verbose`** to see the current 1-vs-1 drift signal — the trend catches slow erosion, but the 1-vs-1 view shows whether the latest cycle is the worst one (which would suggest a recent change) or part of a longer pattern (which would suggest structural drift).
5. **If no root cause matches, escalate per Move #6.10 §Next moves** — the per-cycle alert webhook (`scripts/attribution_health_alert_webhook.py`) is the canonical escalation path; the trend is the slow-burn companion.

## Common pitfalls

15 numbered pitfalls — each with a corrective `Fix:` line clustered into 5 failure modes:

### A. Threshold-loosening (the canonical anti-pattern)

**Pitfall #1 — Loosening `fire_on_cumulative_match_rate_drift_pp` to make alerts stop firing.**
The whole point of the trend script is to catch slow erosion that 1-vs-1 misses. Loosening the cumulative threshold to ≥5pp re-creates the Move #6.8 blind spot (1-vs-1 drift) on a different timescale.
**Fix:** NEVER loosen the canonical thresholds without a documented reason AND a `TICK: <date>` marker in `docs/journal.md`. The thresholds are pinned by `test_canonical_trend_thresholds_published` (4 sub-tests). If the alerts are noise, the fix is upstream: investigate WHY the cumulative drift is climbing (the 5-canonical-root-cause ladder in Step 6) — don't paper over it with a threshold bump.

**Pitfall #2 — Loosening `fire_on_consecutive_decline_weeks` from 4 to 2 to catch earlier signals.**
A 2-week streak is normal noise (one bad week + one good week happens 20% of the time). The 4-week threshold is calibrated to filter out that noise.
**Fix:** keep the 4-week threshold; if you genuinely need earlier signal, add an OVERLAY alert at `--cooldown-seconds 86400` (daily cadence) with a 2-week streak threshold, but treat those as `info` not `warning`. The 4-week `warning` remains the canonical signal.

**Pitfall #3 — Removing `cooldown_seconds` because "trend fires every Monday".**
The 86400-second (24-hour) cooldown is what prevents the Monday-cron from firing the same alert twice if it re-runs Tuesday-Friday (e.g. if a partner manually re-runs the script for verification).
**Fix:** keep `cooldown_seconds=86400`; if you need multi-cycle investigation, use `--skip-cooldown` ONLY for ad-hoc operator runs, NEVER in the cron entry.

### B. Rollup-source drift (substrate corruption)

**Pitfall #4 — Reading the trend script's output as if it were the Move #6.8 rollup output.**
The trend payload has `per_platform_trend.<platform>.match_rate.direction` + `cumulative_drift` — NOT `gates[].value` like Move #6.8. Downstream consumers parsing the trend payload need a different schema.
**Fix:** when piping the trend payload to a downstream consumer (e.g. a dashboard renderer or Linear ticket creator), check the `source` field of the payload — `"attribution_weekly_rollup_trend"` vs `"attribution_cross_platform_rollup"`. Use distinct parsers; never assume the same shape.

**Pitfall #5 — Pointing `--rollup-dir` at the Move #6.8 fixture dir BEFORE Move #6.8 has actually written ≥4 files.**
With fewer than 4 rollup JSONs, the consecutive-decline rule cannot fire (streak counter needs ≥4 weeks). The script will still run and report `weeks_analyzed: N` honestly, but the operator may expect alerts that never come.
**Fix:** confirm `--rollup-dir` has ≥4 Move #6.8 JSONs (use `ls ./fixtures/rollups/ | wc -l`); if <4, the canonical first month is `info`-only — no fire expected, no investigation needed.

**Pitfall #6 — Reading rollup JSONs with the wrong filename pattern.**
The trend script's `_list_weekly_rollups()` walker looks for `weekly-*.json` + `rollup-*.json` + fallback `*.json` in chronological order. If your Move #6.8 cron names files `cross_platform_2026-07-06.json` (not matching any pattern), the walker falls through to `*.json` and may pick up unrelated JSONs in the dir.
**Fix:** canonicalize Move #6.8's output filename to `weekly-<ISO-date>.json` (e.g. `weekly-2026-07-06.json`); update Move #6.8's `--output-json` flag in the cron entry. The pattern-matching walker picks up the canonical pattern first.

### C. Cron-timing (substrate freshness)

**Pitfall #7 — Running the trend script BEFORE Move #6.8 has finished writing the latest JSON.**
Move #6.8 typically finishes in <2 minutes for solo/small-team operator workloads, but on a heavy weekly cycle (4+ platforms, multiple fixture dirs) it can take 5+ minutes. The trend script reading a half-written JSON returns a malformed-rollup sentinel (`None`) and excludes that week from the trend — silently shrinking the window by 1.
**Fix:** schedule the trend cron at least 15 minutes AFTER Move #6.8's cron start time (canonical: Move #6.8 Monday 09:00, trend Monday 09:15). For larger teams running Move #6.8 daily, the trend can run daily too — same 15-minute buffer applies.

**Pitfall #8 — Running the trend script on a host that cannot read the Move #6.8 rollup dir.**
If Move #6.8 writes to `/var/log/attribution-rollup-latest.json` (single-file mode) but the trend script reads `./fixtures/rollups/*.json` (multi-file mode), the trend walker finds 0 files.
**Fix:** canonical Move #6.8 mode is `--output-json ./fixtures/rollups/weekly-<date>.json` (one file per cycle, multi-file mode). Update Move #6.8's cron entry to use the multi-file pattern. Single-file mode loses historical trend capability — Move #6.8's per-cycle dashboard does NOT need history, but the trend script DOES.

**Pitfall #9 — Running the trend script on the wrong week's "latest" file.**
If Move #6.8 overwrites the same filename every week (e.g. always `latest.json`), the trend walker finds only 1 file and reports `weeks_analyzed: 1`. The cumulative window is meaningless.
**Fix:** canonical Move #6.8 cron writes date-stamped files: `--output-json ./fixtures/rollups/weekly-$(date -I).json`. The trend walker picks them up in chronological order.

### D. Slack-format (downstream consumer confusion)

**Pitfall #10 — Sending `info` severity alerts to a noisy channel where operators mute them.**
The trend script fires on EVERY weekly cycle (`info` when stable, `warning`/`critical` when drifting). If the Slack channel is also the per-cycle Move #6.10 alert channel, operators will mute it and miss the `critical` trend fires.
**Fix:** route the trend alerts to a SEPARATE Slack channel (canonical: `#attribution-trend-alerts`). The Move #6.10 per-cycle alerts stay on `#attribution-rollup-alerts` (the existing canonical channel). The two channels have different cadence (daily vs weekly) and different urgency (per-cycle drift vs slow-erosion trend).

**Pitfall #11 — Using a custom Slack format that drops the `remediation` field.**
The canonical 15-field payload has `remediation` — the operator-actionable next-step text. If your Slack template drops it, the operator sees a per-platform trend table but no remediation.
**Fix:** use the canonical Slack template from Step 5; the `remediation` field is non-optional. If your team wants a different format, customize the wrapper but ALWAYS preserve `remediation` + `per_platform_trend` + `cumulative_drift` + `consecutive_decline_platforms`.

**Pitfall #12 — Failing to pin the canonical payload shape in a downstream dashboard.**
If you build a `dashboards/attribution-weekly-rollup-trend.html` (a future tick), the dashboard MUST read the canonical 15-field shape — not a custom shape. Drift between the script's payload and the dashboard's parser is a regression.
**Fix:** the next-tick agent should reference `_canonical_trend_alert_shape_published()` (which lists the 15 fields) when building the dashboard. The dashboard should use a `function extractTrendPayload(archiveFile)` helper that validates the field set against the canonical list BEFORE rendering.

### E. Hermetic-archive fallback (sub-canonical-paths)

**Pitfall #13 — Setting `--alert-archive` to a path the script cannot write to.**
The hermetic archive fallback is canonical — without it, the script has no persistence. If `--alert-archive` points to a read-only directory (e.g. `/etc/trend-alerts/`), the script silently fails the archive write and the operator has no record.
**Fix:** the canonical default is `./.trend-alerts/` (relative to cwd). For production, use `/var/log/trend-alerts/` (canonical for systemd-housed deployments) or `~/.trend-alerts/` (canonical for operator-laptop ad-hoc runs). Verify write access with `touch /var/log/trend-alerts/.write-test && rm /var/log/trend-alerts/.write-test` before the first cron run.

**Pitfall #14 — Setting `--alert-archive` to a path inside the git working tree.**
The archive directory contains one JSON per cycle — over a year, that's 52 files in `./.trend-alerts/`. If you forget to add `.trend-alerts/` to `.gitignore`, every `git add` picks up the archive.
**Fix:** add `.trend-alerts/` to `.gitignore` immediately after `--bootstrap`. Canonical `.gitignore` line: `.trend-alerts/`.

**Pitfall #15 — Mixing Move #6.10 archive files with Move #6.5 trend archive files in the same directory.**
Both scripts use the same `cooldown_seconds=86400` walker pattern, but Move #6.10 archive files are `alert-<timestamp>-<severity>.json` while Move #6.5 trend files are `trend-<timestamp>-<severity>.json`. If they share a directory, the cooldown walker reads both — which is harmless (the walker filters by filename prefix), but operators grepping the dir for "alert" miss the trend files.
**Fix:** use distinct directories — canonical Move #6.10 archive is `./.alerts/` and canonical Move #6.5 trend archive is `./.trend-alerts/`. The two scripts share NO state.
