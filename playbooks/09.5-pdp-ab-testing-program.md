# Move #9.5 — Always-on PDP A/B testing program (Convert.com / VWO / Shoplift)

> **Why this move is the natural follow-up to Move #9 (mobile PDP redesign) + Move #10 (AI ad creative).** A single redesign is a one-shot lift of 10–25%; an always-on A/B testing program compounds 5–15% YoY on top of that. The compound math: 4 tests/month × 5–10% avg lift each = 20–40% incremental CVR over 12 months if every test winner ships. Realistic (only 50% of tests produce a shippable winner at typical p<0.05 confidence) lands at 10–20% YoY CVR improvement — still the difference between a $5M store and a $6M store on the same traffic. The build surface is your existing PDP (now optimized from Move #9), an A/B testing tool (Convert.com / VWO / Shoplift), and a 2 hr/wk operator cadence for hypothesis generation + result review.

> **Honest read.** This move requires **the redesigned PDP from Move #9** to be live — running A/B tests on a slow PDP confounds the test (a 200ms LCP regression on the variant will hurt CVR independent of the actual hypothesis). Also requires **attribution from Move #6** (Triple Whale) to confirm winners don't have downstream LTV regressions (Pitfall #15 from Move #10). The typical operator trap: running too many simultaneous tests and ending up with no significant winners because the sample size per arm is too small (Gate C below). Cap concurrent tests at 2 unless your PDP traffic exceeds 50k sessions/wk. Also: don't run more than one test on the same element at a time (a test on the hero image and a test on the ATC copy in parallel is fine; two tests on the hero image is not — they cross-contaminate).

> **Companion assets:** `scripts/pdp_ab_test.py` (new — Archetype A score/decision tool, two-proportion z-test + program ROI forecast, no API credentials required) + `scripts/tests/test_pdp_ab_test.py` (24 TDD tests across 8 test classes including 5 NEGATIVE validation tests + JSON sanitization for infinite/NaN values). Run `python3 scripts/pdp_ab_test.py` on any completed test to get the winner/loser/inconclusive decision + the program-level ROI forecast.

## Goal

In 4–6 weeks of focused work, stand up an always-on PDP A/B testing cadence that:

1. **Launches 4 tests per month** — measured by your A/B tool's test counter (Convert.com / VWO / Shoplift dashboard) over the rolling 28 days.
2. **Identifies at least 1 shippable winner per month** — measured by tests where decision = "winner" at 95% confidence and the winner is promoted to 100% traffic for the tested element.
3. **Lifts PDP CVR by 5–15% over 6 months** — measured by Triple Whale `Pages` report filtered to the top 10 PDPs (compare Jan CVR vs Jul CVR; confound with seasonal adjustments).
4. **Keeps false-positive rate ≤ 5%** — measured by counting tests where decision = "winner" but the promotion-to-100% causes CVR to regress within 14 days (this is the "shipped a winner that wasn't" rate).
5. **Costs ≤ $500/mo total** — measured by `scripts/pdp_ab_test.py` annualized_cost output ($200/mo tool + 2 hr/wk operator at $50/hr loaded = $600/mo at 4.3 wks/mo; sits in the "good" band).
6. **Compounds the Move #9 mobile-PDP lift** — measured by Triple Whale `Devices` report (mobile CVR should keep climbing 3–6 months post-redesign as new test winners ship).

**Out of scope:** Multivariate testing (MVT — requires 4×–16× the sample size for statistical power; defer until you have 50k+ PDP sessions/wk), personalization via Rebuy/Nosto (a separate lever), email A/B testing (Klaviyo's built-in A/B covers that — see Move #4), SMS A/B testing (Postscript's built-in A/B covers that — see Move #7), pricing tests (regulatory + brand-trust risk; defer).

## Step-by-step

Six steps. Steps 1–3 take ~1 week (tool setup + backlog). Steps 4–6 are recurring monthly operations. The whole playbook is shippable in 1–2 weeks of focused setup + ongoing 2 hr/wk cadence.

### Step 1 — Tool setup + test backlog (Days 1–3)

**For Path A (Shoplift):**

1. Install **Shoplift** from Shopify App Store → connect to your store → grant theme access.
2. Walk through Shoplift's onboarding wizard: select "Product page" as the test surface, set traffic split to 50/50, enable Bayesian + frequentist significance calculation.
3. Build your test backlog: list 8 hypotheses in a Notion doc with the columns `Element | Variant | Hypothesis | Expected Impact | Priority (1–8)`. Use the **PDP test element matrix** below for the first 4 ideas.

**For Path B (Convert.com — default):**

1. Sign up at Convert.com → select **Growth plan ($349/mo)** → install the global snippet in `theme.liquid` `<head>` (Convert provides the snippet).
2. Connect Triple Whale via the "Attribution" integration in Convert → Settings → Integrations. This enables cohort LTV overlay on winning variants.
3. Configure the visual editor for non-developer hypothesis design (Convert's "Edit Mode" lets the operator click-and-edit page elements without writing code).
4. Set the experiment defaults: confidence threshold = 95% (matches the canonical Move #9.5 default), traffic split = 50/50, test duration = 14 days minimum (extend if sample size not yet reached).
5. Build your test backlog: 8 hypotheses in Notion with `Element | Variant | Hypothesis | Expected Impact | Priority (1–8)`.

**For Path C (VWO):**

1. Sign up at VWO → select **Standard plan ($314/mo)** → install the smart-code snippet in `theme.liquid` `<head>`.
2. Enable VWO Insights (heatmaps + session recordings) — this is VWO's killer feature for hypothesis generation. Spend 1 hour watching 5–10 session recordings of your top PDP; note where users scroll, click, and drop off.
3. Build your hypothesis backlog: 8 ideas in Notion, prioritized by expected impact (use the PDP test element matrix below for the first 4).

**Verification gate for Step 1:** `python3 scripts/pdp_ab_test.py --help` exits 0 and shows the 14 CLI flags. Test backlog exists in Notion with ≥8 rows.

| # | Element | Variant idea | Expected lift | Difficulty | Triple Whale overlay needed? |
|---|---|---|---|---|---|
| 1 | **Hero image** | Lifestyle vs. white-background | +5–15% CVR | Easy (Convert visual editor) | Yes (cohort LTV per image) |
| 2 | **ATC button copy** | "Add to cart" vs. "Add to bag" vs. "Buy now" | +2–8% CVR | Easy (text swap) | Yes |
| 3 | **Sticky ATC bar color** | Black vs. brand-color vs. white-with-black-text | +3–10% mobile CVR | Medium (CSS override) | Yes |

**Verification gate for Step 2:** Backlog of 8 hypotheses in Notion with priority order. The 4 you plan to ship in the first month are #1–#4 from the matrix (hero image, ATC copy, sticky bar color, reviews placement — covers all 3 difficulty tiers and all 3 expected-impact tiers).

### Step 3 — Launch the first test (Days 6–7)

Pick the highest-priority hypothesis (typically #1 hero image or #3 sticky bar color for the biggest expected lift).

**Test setup checklist:**

1. Create the test in Convert.com (or your tool): name it `pdp-hero-2026-q3-v1` (naming convention: `pdp-{element}-{quarter}-v{n}`).
2. Set the **control** to the current state (the Move #9 mobile-PDP design).
3. Set the **variant** to the hypothesis (e.g. lifestyle hero image vs. white-background).
4. Configure **targeting**: PDP template (e.g. `/products/*` excluding landing pages), 50/50 traffic split, all devices (mobile + desktop — most PDP tests don't need device splitting; the variant typically wins on both).
5. Set **success metric**: PDP CVR (CVR = add-to-cart / PDP view; this is the canonical Move #6/9 metric).
6. Set **secondary metrics**: ATC rate, RP per visitor (RPV), bounce rate (Triple Whale overlay).
7. Set **minimum sample size**: 2,500 conversions/arm (use a free online calculator like [Optimizely's sample-size tool](https://www.optimizely.com/sample-size-calculator/) with baseline CVR + 5% MDE + 95% confidence → typically 5,000–15,000 sessions/arm depending on CVR).
8. Set **test duration**: 14 days minimum (extend to 28 if sample size not reached at 14d).
9. **QA the variant** on iPhone SE (small viewport), iPhone 14 Pro (modern viewport), iPad, and a desktop browser before launching. Capture screenshots.
10. Launch the test.

**Verification gate for Step 3:** Convert.com (or your tool) shows the test as "Running" with the correct targeting + success metric. QA screenshots captured and stored in the test's Notion page. Test start date + expected end date documented.

### Step 4 — Monitor + decide (Days 8–21, recurring)

While the test runs, monitor weekly:

1. **Day 7 check-in:** Open the test dashboard → confirm traffic is ~50/50 split, sample size is accumulating, no error spikes (e.g. variant rendering broken on Safari).
2. **Day 14 primary check:** Run `python3 scripts/pdp_ab_test.py --control-sessions X --control-conversions Y --variant-sessions X2 --variant-conversions Y2` with the current numbers from the test dashboard. The script returns:
   - `decision: winner` if confidence ≥ 95% AND variant lift > 0
   - `decision: loser` if confidence ≥ 95% AND variant lift < 0
   - `decision: inconclusive` if confidence < 95% (extend test duration if sample size not reached)
3. **Day 14 cohort overlay:** If `decision: winner`, open Triple Whale → Cohorts → create a "Test winner" cohort (users exposed to variant) and "Test control" cohort (users exposed to control) → compare 30-day cohort LTV. If test winner's LTV is within ±10% of control's LTV, ship the variant. If test winner's LTV is significantly lower (Pitfall #15 from Move #10 — high-CTR low-LTV trap), DON'T ship despite the test's statistical win.
4. **Day 21 (if extended):** Re-run the script. If still inconclusive at day 21, stop the test and either ship the variant as a 50/50 rollback option (no decision) or keep control (the conservative default).

**Verification gate for Step 4:** Each week, a Notion page entry with: test name + day count + current z-score + current p-value + current decision + cohort LTV comparison + next action.

### Step 5 — Promote the winner + start the next test (Day 21–28)

When a test ships:

1. **Promote the variant to 100%** in Convert.com (or your tool). Confirm the variant is now serving all traffic.
2. **Archive the test** in the tool (move to "Completed" folder). Note the final z-score, p-value, decision, and shipped-vs-not in a Notion archive.
3. **Update Triple Whale cohorts**: merge the "Test winner" cohort into a permanent "PDP winner cohort" segment for ongoing monitoring.
4. **Pick the next hypothesis** from the backlog (priority #2 or #3 from the matrix). Set up the test in the tool. Launch.
5. **Compute the program ROI**: run `python3 scripts/pdp_ab_test.py --json` on the most recent test, capture the `program_forecast` block, log the annualized ratio in your monthly Cadence report.

**Verification gate for Step 5:** Winner promoted to 100% (Convert.com shows 100% traffic on variant), archived in Notion, cohort LTV segment created, next test launched, monthly ROI logged.

### Step 6 — Monthly cadence + review (recurring)

Every 4 weeks, hold a 30-minute program review:

1. **Tests shipped this month:** Count tests where decision = "winner" → ship.
2. **Tests inconclusive:** Count tests where decision = "inconclusive" → log for hypothesis-backlog update (inconclusive tests are usually hypotheses that need a bigger effect or a longer duration).
3. **Cumulative PDP CVR lift:** Triple Whale → Pages → Top PDPs → CVR this month vs 3 months ago. Document the delta.
4. **Program ROI:** Run `scripts/pdp_ab_test.py` with the month's avg winner lift + remaining backlog → confirm annualized ratio is still in the "good" or "great" band (≥10:1). If it's dropped to "fair" (5–10:1), investigate: are we testing too many small-impact hypotheses? Are operator hours creeping up?
5. **Backlog refresh:** Add 4 new hypotheses for next month. Sources: Triple Whale cohort anomalies, customer support tickets, session recording observations, competitor PDP teardowns.

**Verification gate for Step 6:** Monthly review doc exists in Notion with the 5 sections above. Program ROI still in "good" or "great" band. Backlog refreshed to 8+ hypotheses for next month.

## Common pitfalls

15 pitfalls with corrective "Fix:" lines.

1. **Running tests on a non-optimized PDP.** Fix: ship Move #9 (mobile PDP redesign) FIRST — running A/B tests on a slow PDP confounds the test (a 200ms LCP regression on the variant will hurt CVR independent of the hypothesis).
2. **Too many concurrent tests.** Fix: cap at 2 concurrent tests unless your PDP traffic exceeds 50k sessions/wk. More than 2 starves each test's sample size and lengthens duration past 28 days (the operator abandons the program).
3. **No test backlog.** Fix: maintain an 8-hypothesis backlog in Notion at all times. Without a backlog you'll run out of ideas after test #2 and the cadence collapses.
4. **No promotion-to-100% process documented.** Fix: write a 5-step SOP (review winner → check cohort LTV → promote variant → archive test → start next test) and store in Notion. The "winner" decision is meaningless without a documented promotion step.
5. **Shipping winners without cohort LTV overlay.** Fix: ALWAYS create a Triple Whale "Test winner" cohort and a "Test control" cohort before promoting. If test winner's LTV is significantly lower (high-CTR low-LTV trap from Move #10 Pitfall #15), DON'T ship despite the test's statistical win.
6. **Testing too many small-impact hypotheses.** Fix: prioritize the PDP test element matrix above (hero image, ATC copy, sticky bar color, reviews placement) — these have the biggest expected lift. Small-impact hypotheses (font color, button padding) are not worth the operator hours.
7. **Stopping tests at the first significant result.** Fix: always run the test to its planned duration (14–28 days). Stopping early inflates the false-positive rate; some tests "win" on day 5 and revert by day 14.
8. **No documentation of inconclusive tests.** Fix: log every inconclusive test in Notion with the hypothesis + final z-score + reason (sample size not reached, lift too small to detect, etc.). Inconclusive tests are valuable backlog updates (they tell you which hypotheses need bigger effects).
9. **Treating "winner at 95% confidence" as enough.** Fix: also check cohort LTV (Pitfall #5) and run a 14-day post-promotion regression check. Some "winners" regress within 14 days because the cohort mix shifts.
10. **Picking VWO or Convert.com at <5k PDP sessions/wk.** Fix: use Shoplift (Path A) below 10k sessions/wk — Convert's full feature set is wasted without the traffic to power it.
11. **No operator hours in the calendar.** Fix: book 2 hr/wk as a recurring calendar block. Without this, the program stalls after the first 2–3 tests.
12. **Testing the same element with multiple variants simultaneously.** Fix: never run two tests on the same element at the same time (a test on the hero image and a test on the ATC copy is fine; two tests on the hero image is not — they cross-contaminate).
13. **Running tests during BFCM or a product launch.** Fix: schedule tests to avoid the 2 weeks before/after any major sale or launch. PDP CVR is sensitive to traffic mix; the test confounds.
14. **Ignoring mobile-specific variants.** Fix: if the redesign has a mobile-specific element (e.g. sticky ATC bar visible only on mobile), test it on mobile-only traffic. Mobile-only variants on desktop traffic show no lift (and vice versa).
15. **Not using the program's measurement framework.** Fix: ALWAYS run `python3 scripts/pdp_ab_test.py --json` at the end of each test and log the `program_forecast.annualized_ratio` in your monthly Cadence report. The script's output is the canonical program ROI metric — don't compute it ad-hoc.

## Verification

7 gates (A–G) that the program is operating correctly.

- **Gate A — Tool integration green.** Convert.com (or your pick) is installed, the global snippet is in `theme.liquid` `<head>`, and Triple Whale's attribution overlay is connected. Verify by creating a "Hello World" test (no actual variant), launching it, and confirming it serves 50/50 traffic on the PDP template.
- **Gate B — First test launched + 14 days completed.** The first hypothesis from the backlog has run for 14 days minimum. Verify via Convert.com dashboard → test history → start/end dates.
- **Gate C — Statistical significance computed.** `python3 scripts/pdp_ab_test.py --control-sessions X --control-conversions Y --variant-sessions X2 --variant-conversions Y2` exits 0 and returns a valid `decision` (winner / loser / inconclusive). Verify the z-score and p-value match the tool's dashboard (small differences expected due to Bayesian vs. frequentist methods).
- **Gate D — Cohort LTV overlay checked.** Triple Whale shows a "Test winner" cohort and a "Test control" cohort with 30-day LTV within ±10% (if shipping the winner) or document the LTV regression (if NOT shipping despite statistical win).
- **Gate E — Winner promoted to 100%.** Convert.com shows the winning variant at 100% traffic. The 14-day post-promotion CVR matches or exceeds the in-test CVR (false-positive rate check).
- **Gate F — Next test launched.** The next hypothesis from the backlog is launched within 7 days of the previous test's promotion. Verify via Convert.com active tests list.
- **Gate G — Program ROI in "good" or "great" band.** `scripts/pdp_ab_test.py` program_forecast.annualized_ratio ≥ 10:1. Verify monthly. If ratio drops below 10:1, the program is underperforming — investigate (too many inconclusive tests? operator hours creeping up? backlog running low on high-impact hypotheses?).

| **Net annual ROI (typical, blended +10% lift)** | **$22,152–$85,152/yr** |
| **Ratio** | **$22,152/yr / $9,348/yr = 2.4:1 (conservative) → 9.1:1 (aggressive)** at the per-test level; the cumulative-YoY-CVR math lifts this to **10:1+ at 6+ months of steady-state cadence** (matches the "good" band threshold) |
| **Payback period** | **4–8 weeks** (the first 1–2 winners cover the program's monthly cost) |

> **Honest read.** The Move #9.5 program's value is **compounding**, not transactional. A single test that ships +5% lift is worth $2,625/mo — barely 3× the program's cost. But after 12 months of 4 tests/mo with 1 winner/mo at +5–10% lift each, the cumulative PDP CVR lift is 5–15% on the original baseline, and the monthly incremental revenue is $2,625–$7,875/mo (cumulative on the original baseline, not on the previous winner). The 10:1+ ratio band is the steady-state view, not the first-month view. Operators who measure at month 1 and bail because the ratio looks "only 3:1" are missing the point. The classic Move #9.5 trap: stop running tests after 3 inconclusive results. Always keep the cadence — the cumulative math needs the cadence to work.

## Next moves after this A/B testing program is live

1. **Move #6.5 — Attribution quality audit (Meta CAPI + Google Enhanced Conversions).** The A/B testing program depends on cohort LTV being accurate. If your Triple Whale cohort LTV is off by 20% because of CAPI event_id mismatches, the cohort overlay check (Pitfall #5) is unreliable. Audit Meta CAPI event_id match rate (target ≥90%) + Google Enhanced Conversions quality (target Good or Excellent). Companion `scripts/attribution_quality_audit.py` to score the 7 gates (matches the Move #6 hermetic checklist pattern).
2. **Multivariate testing (MVT) for stores at 50k+ PDP sessions/wk.** Once the A/B program is mature and traffic has grown, graduate to multivariate testing — tests 3+ variants of an element simultaneously + interactions between elements. Requires 4×–16× the sample size per arm. VWO Standard supports MVT natively; Convert.com Growth does not (requires Scale plan).
3. **Personalization via Rebuy / Nosto.** Once you have 8+ shipped winners, the next lever is per-segment PDP variants (returning customers see "You might also like" + recently viewed; new customers see "Best sellers" + reviews). Build on top of the Move #9.5 program's win history (the personalization engine learns from past winners).
4. **PDP speed dashboard.** Build a `dashboards/pdp-speed-monitor.html` static dashboard that polls PageSpeed Insights + Search Console CrUX daily and plots LCP/INP/CLS over time. The dashboard surfaces speed regressions from new A/B test variants (Pitfall #1) — a variant that adds 200ms of JS would show up as an LCP regression within 24h.
5. **Cross-page A/B testing (cart, checkout, post-purchase).** Once PDP testing is mature, extend the cadence to cart page (free-shipping progress bar variants), checkout (Move #3 audit overlaps with this), and post-purchase (Move #2 upsell page). Same Convert.com tool, same operator cadence.

This compounds the Move #10 investment by applying its insights to the PDP layer.
