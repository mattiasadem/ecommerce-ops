# Move #6.5 — Attribution quality audit (Meta CAPI + Google Enhanced Conversions + GA4 ↔ Triple Whale)

> **Why this move is the foundation that every downstream lever depends on.** Move #6 installs Triple Whale (or Polar). Move #7 (SMS), Move #8 (loyalty), Move #9 (mobile PDP), Move #9.5 (PDP A/B testing), and Move #10 (AI ad creative) all reference Triple Whale's cohort LTV in their verification gates — "the cohort overlay check", "the killer test", "promote to 100% only if LTV is preserved". Those checks are meaningless if the underlying attribution is silently degraded. Meta's own documentation says CAPI match rates below 80% will quietly hurt campaign optimization; below 70% Meta throttles CAPI delivery entirely. Google Enhanced Conversions quality "Unavailable" means Google is falling back to coarse city/state matching and your Smart Bidding is flying blind. GA4 vs Triple Whale revenue drifting more than 5% post-iOS14.5 is the classic "two dashboards disagree" trap that makes operators abandon the tool after 30 days. **This audit catches all of these before they cost you a quarter of bad spend decisions.**

> **Honest read.** Move #6's companion script (`scripts/triple_whale_attribution_check.py`) validates **installation** — does the pixel exist, are the keys set, are the flow names correct. Move #6.5's companion script (`scripts/attribution_quality_audit.py`, new this tick) validates **measurement quality** — are the events matching, is the dedup working, is the quality tier high enough, are the dashboards reporting the same numbers. The two are sequential: Move #6 must pass all 7 gates first, THEN run Move #6.5 to confirm the install is producing trustworthy signal. A Move #6 pass with a Move #6.5 fail is the most common attribution-stack failure mode — the operator trusts Triple Whale's cohort LTV report and makes a $50k ad-spend reallocation decision based on numbers that are 30% wrong.

> **Companion script:** `scripts/attribution_quality_audit.py` (new this tick, ~700 lines) is a hermetic measurement-quality audit that validates 6 numeric-quality gates against JSON fixtures the operator exports from each platform's diagnostic report. It does NOT require API credentials and cannot authenticate against any platform — instead, it reads the local diagnostic-export JSONs (Meta Events Manager → Diagnostics, Google Ads → Conversions → Diagnostics, GA4 → Monetization, Triple Whale → Revenue, Klaviyo → Customers) and runs the same 6 quality gates the playbook documents. Default bootstrap generates canonical-pass fixtures so the operator can run the audit immediately and see all-green.

## Goal

In a single 2–3 hour session, stand up a weekly measurement-quality audit cadence that:

1. **Confirms Meta CAPI event_id match rate is ≥ 90%.** Meta's own docs: below 80% silently degrades campaign optimization; below 70% Meta throttles CAPI delivery. The audit runs weekly against the previous 7-day Meta diagnostic export. The threshold is published in `MIN_META_CAPI_MATCH_RATE_PCT = 90.0` in the script and pinned by a TDD test.
2. **Confirms Meta CAPI dedup ratio is in [0.8, 1.5].** `pixel_events / capi_events` should be 0.8–1.5 — outside that band means one side is over-firing (double-counting risk) or under-firing (missing attribution). The threshold is the canonical "dedup is real" check; both bounds are pinned by tests.
3. **Confirms Meta Pixel coverage is ≥ 95% of expected pageviews.** Below 80% means the pixel is not on every page (usually a theme.liquid edit was missed during a theme update); 80–95% is normal ad-blocker-induced loss.
4. **Confirms Google Enhanced Conversions quality tier is "Good" or "Excellent"** AND **hashed-email coverage is ≥ 80%**. The email-hashing pipe is the EC killer — without it Google falls back to coarse city/state matching and EC quality is "Unavailable." Below 80% email coverage means guest checkouts aren't being captured.
5. **Confirms GA4 ↔ Triple Whale revenue delta is ≤ 5% in the 7-day window.** Move #6 Pitfall #15: GA4 misses 30–40% post-iOS14.5. Triple Whale is the source of truth (it merges pixel + CAPI + survey), so a 5%+ delta with GA4 means GA4 is undercounting (expected behavior, document it) OR TW is overcounting (refund/return miscategorization, investigate). A 0% delta with GA4 is more suspicious than a 5% delta — that means Triple Whale is reading the same data as GA4 and the pixel+CAPI merge isn't actually working.
6. **Confirms Klaviyo ↔ Triple Whale cohort roundtrip is ≥ 95% match rate** on at least 5 sample orders. The cohort sync is "configured" once the Move #6 Gate C passes; this audit confirms the sync is actually firing on real orders (not just configured and silently broken).

**Out of scope:** Northbeam / Rockerbox / Beeyond MTA (premium tier — they run their own measurement quality audit at the platform level; out of scope for Move #6.5), MMM (Media Mix Modeling — Triple Whale Pro or higher; their quality audit is different from the per-platform CAPI/EC checks), in-house attribution engineering, custom data-warehouse pipelines.

## Which audit cadence fits your team

Pick one of three cadences. The decision is driven by **team size + ad spend velocity + how often you make attribution-dependent decisions**, not by how thorough you want to be.

| Path | Team profile | Audit cadence | Tool | Why this one |
|---|---|---|---|---|
| **A** | Solo operator, <$5k/mo paid spend, ships attribution decisions monthly | **Monthly** audit, run on the 1st of each month | `scripts/attribution_quality_audit.py` (manual invocation) | Cheapest cadence. Monthly catches drift between quarters; the decision cycle is monthly so more frequent audits are noise |
| **B** | Small team (2–5 people), $5k–$50k/mo paid spend, ships attribution decisions weekly | **Weekly** audit, run Monday morning | Same script, plus a cron job that emails the JSON output | The default for Move #6 shops. Weekly catches the common CAPI match-rate regressions within 1 decision cycle; emails the JSON to a Slack channel for visibility |
| **C** | Larger team (5+ people), $50k+/mo paid spend, ships attribution decisions daily | **Daily** audit, run at 06:00 UTC | Same script, plus a Vercel cron or AWS EventBridge that calls the script and pushes the JSON to a dashboard | Above $50k/mo the cost of a single bad decision is meaningful (one ad-set with bad attribution can burn $500/day). Daily catches issues within 24h |

**Default for this playbook: Path B (weekly audit, Monday morning).** Path A for solo operators who don't need weekly visibility. Path C only for stores spending $50k+/mo where 24h regression detection matters.

**Decision rule of thumb:** if your paid spend is ≥$5k/mo AND you have ≥2 attribution-dependent decisions per month (reallocate budget between channels, launch a new campaign, kill an underperformer), Path B is the minimum. Below $5k/mo spend the decision cycle is too slow for weekly audits to add value.

## Step-by-step

Six steps. Steps 1–3 are the one-time setup (~1 hour). Steps 4–6 are the recurring weekly cadence (~30 min per audit). The whole playbook is shippable in 2–3 hours of initial setup + ongoing 30 min/wk.

### Step 1 — Export the 6 diagnostic JSON files (one-time, ~30 min)

For each platform, navigate to the diagnostic export, copy the values into the JSON fixture template, and save the file to `./attribution_fixtures/`. The script's `--bootstrap` flag generates a starter fixture set in the right directory.

**Meta Events Manager diagnostic (last 7d):**

1. Go to events.manager.facebook.com → select your pixel → "Diagnostics" tab → "Event Quality" subtab.
2. Set the time range to "Last 7 days."
3. Note four numbers: `Pixel events` (count of `Purchase` events fired by the browser pixel), `Server events` (count of `Purchase` events fired by CAPI), `Matched events` (count of `Purchase` events matched by `event_id`), and `Expected orders` (count of `Order created` webhooks from Shopify in the same window — get this from `Shopify → Analytics → Total orders` filtered to last 7d).
4. Paste these into `attribution_fixtures/meta_capi_match.json`:
   ```json
   {
     "pixel_events": 1000,
     "capi_events": 950,
     "matched_events": 920,
     "expected_orders_last_7d": 950
   }
   ```
5. Separately, for Gate C, get `PageView event count` from the same Diagnostics tab → "Event Quality" → filter to `PageView` events → note the count. Get `Total sessions` from `Shopify → Analytics → Total sessions` for the same 7d window. Paste into `attribution_fixtures/meta_pixel_coverage.json`.

**Google Ads Enhanced Conversions diagnostic (last 7d):**

1. Go to ads.google.com → Tools → Conversions → Summary → click the "Purchase" conversion action → "Diagnostics" tab.
2. Note the `Enhanced conversions quality` tier (will be one of "Excellent", "Good", "Needs improvement", or "Unavailable"). Note `Conversions with hashed email` (the count of conversions where the email-hashing pipe fired) and `Total conversions` (the count of all conversion events in the window).
3. Compute `hashed_email_coverage_pct = conversions_with_hashed_email / total_conversions * 100` and paste all four into `attribution_fixtures/google_enhanced_quality.json`:
   ```json
   {
     "quality_tier": "Good",
     "hashed_email_coverage_pct": 85.0,
     "total_conversions": 1000,
     "conversions_with_hashed_email": 850
   }
   ```

**GA4 + Triple Whale revenue (last 7d):**

1. Go to analytics.google.com → Reports → Monetization → Ecommerce purchases → set date range to "Last 7 days." Note `Total revenue` (the GA4-reported revenue from `purchase` events).
2. Go to app.triplewhale.com → Revenue → set date range to "Last 7 days." Note `Revenue` (the TW-reported revenue, which merges pixel + CAPI + survey).
3. Go to Shopify → Analytics → Total orders last 7d (use this as `actual_orders_last_7d`). For `expected_orders_last_7d`, use the same number (Shopify is the source of truth for order count; a discrepancy means a webhook is delayed).
4. Paste all four into `attribution_fixtures/ga4_tw_revenue.json`:
   ```json
   {
     "ga4_revenue_last_7d": 65000.0,
     "tw_revenue_last_7d": 67500.0,
     "actual_orders_last_7d": 950,
     "expected_orders_last_7d": 950
   }
   ```

**Klaviyo ↔ Triple Whale cohort roundtrip (5 sample orders, last 30d):**

1. Go to Shopify → Orders → filter to last 30 days → pick 5 random orders (use `sort by: random` or just pick 5 in the middle of the list).
2. For each `order_id`: open Klaviyo → Customers → search by email or order_id → note the cohort tag (e.g. "Welcome Series", "Loyalty Member", "SMS Welcome").
3. For each `order_id`: open Triple Whale → Orders → search by order_id → note the TW cohort tag.
4. Build a list and paste into `attribution_fixtures/klaviyo_tw_cohort.json`:
   ```json
   {
     "sample_orders": [
       {"order_id": "1001", "klaviyo_cohort": "Welcome Series", "tw_cohort": "Welcome Series"},
       {"order_id": "1002", "klaviyo_cohort": "SMS Welcome", "tw_cohort": "SMS Welcome"},
       {"order_id": "1003", "klaviyo_cohort": "Loyalty Member", "tw_cohort": "Loyalty Member"},
       {"order_id": "1004", "klaviyo_cohort": "Abandoned Cart", "tw_cohort": "Abandoned Cart"},
       {"order_id": "1005", "klaviyo_cohort": "Organic", "tw_cohort": "Organic"}
     ]
   }
   ```

**Attribution drift baseline (Gate G, one-time):**

1. After the first audit, save the current `meta_capi_match.json` and `ga4_tw_revenue.json` numbers as the "previous" baseline.
2. For now, paste placeholder values into `attribution_fixtures/attribution_drift.json`:
   ```json
   {
     "measurement_window_days": 7,
     "drift_threshold_pct": 5.0,
     "current_match_rate_pct": 96.8,
     "previous_match_rate_pct": 95.0,
     "current_revenue_delta_pct": 3.7,
     "previous_revenue_delta_pct": 4.1
   }
   ```
3. Gate G becomes meaningful on the SECOND weekly audit (when you have a "previous" value to compare against).

**Verification gate for Step 1:** all 6 fixture files exist in `./attribution_fixtures/` and each `json.load()`-parses cleanly (`python3 -c "import json; [json.load(open(f'attribution_fixtures/{n}')) for n in ['meta_capi_match.json', 'meta_pixel_coverage.json', 'google_enhanced_quality.json', 'ga4_tw_revenue.json', 'klaviyo_tw_cohort.json', 'attribution_drift.json']]; print('OK')"`).

### Step 2 — Run the audit (one-time + recurring, ~5 min per run)

1. From the project root, run `python3 scripts/attribution_quality_audit.py`. The script reads `./attribution_fixtures/`, runs all 6 gates, and prints a pass/fail report.
2. For automated cadence (Path B/C), wrap the run in a cron job:

   ```bash
   # Weekly audit every Monday 07:00 UTC, email JSON output
   0 7 * * 1 cd /data/workspace/ecommerce-ops && python3 scripts/attribution_quality_audit.py --json | mail -s "Move #6.5 Weekly Audit" ops@yourbrand.com
   ```

   Or push to Slack via webhook (replace `$SLACK_WEBHOOK` with the actual URL):
   ```bash
   0 7 * * 1 cd /data/workspace/ecommerce-ops && python3 scripts/attribution_quality_audit.py --json | curl -X POST -H 'Content-Type: application/json' -d @- "$SLACK_WEBHOOK"
   ```

3. The script exits 0 if all 6 gates pass, 1 if any gate fails. Use the exit code in CI/cron: `python3 scripts/attribution_quality_audit.py || echo "Move #6.5 audit FAILED — investigate"`.

**Verification gate for Step 2:** the first run should print `Summary: 6/6 gates passed` (assuming you populated the fixtures with real diagnostic values that pass the thresholds). If any gate fails, see the Verification section below for the per-gate remediation.

### Step 3 — Set the alert thresholds (one-time, ~10 min)

1. Open the script (`scripts/attribution_quality_audit.py`) and locate the threshold constants at the top of the file:
   ```python
   MIN_META_CAPI_MATCH_RATE_PCT = 90.0
   MIN_DEDUP_RATIO = 0.8
   MAX_DEDUP_RATIO = 1.5
   MIN_META_COVERAGE_PCT = 95.0
   MIN_META_PIXEL_COVERAGE_PCT = 95.0
   ACCEPTABLE_GOOGLE_EC_QUALITY = {"Good", "Excellent"}
   MIN_GOOGLE_EC_HASHED_EMAIL_PCT = 80.0
   MAX_GA4_TW_REVENUE_DELTA_PCT = 5.0
   MAX_ORDER_COUNT_DELTA_PCT = 3.0
   MIN_COHORT_ROUNDTRIP_MATCH_PCT = 95.0
   MIN_SAMPLE_ORDERS = 5
   ```
2. **Do NOT change these defaults on the first run.** The defaults are the industry-canonical thresholds (Meta's own docs, Google's own docs, Move #6 Pitfall #15). Loosening them without a documented reason is the most common Move #6.5 failure — operators loosen the threshold to make the gate pass, then the next quarter's bad-attribution decision goes unnoticed.
3. **If you must loosen a threshold** (e.g. your store is at 1k orders/mo and the 5% revenue delta is too tight for low-volume variance), document WHY in a comment next to the constant and add a `TICK: <date>` marker so the next operator can revisit it when the store scales.

**Verification gate for Step 3:** the `test_canonical_thresholds_published` test in `scripts/tests/test_attribution_quality_audit.py` pins the threshold values and will fail if they're changed. If you genuinely need to change a threshold, update both the constant and the test in the same commit.

### Step 4 — Investigate any gate failures (recurring, ~15 min per failure)

The audit script's `remediation` field is the entry point for investigation. Common failure patterns:

| Gate | Symptom | Most likely cause | Fix |
|---|---|---|---|
| **A** (CAPI match rate) | `Match rate 78.0% < 90%` | `event_id` mapping broken between pixel and CAPI; pixel not on every page; CAPI token rotated without re-deployment | Verify event_id is identical in pixel + CAPI payloads (Triple Whale → Settings → CAPI → event_id mapping). Check theme.liquid for the pixel. Verify CAPI token in Triple Whale → Settings → Install. |
| **A** (dedup ratio) | `Dedup ratio 2.10 outside [0.8, 1.5]` | Pixel over-firing (double-firing on the same event) OR CAPI under-firing (server is dropping events) | Look for double-firing in Meta Events Manager → Test Events. If pixel > capi: check theme.liquid for duplicate pixel snippet. If capi > pixel: check that CAPI is firing on `order.created` webhook, not on a delayed event. |
| **C** (pixel coverage) | `Pixel coverage 67.0% < 95%` | Pixel snippet removed during a theme update; theme.liquid not deployed to live theme; CSP header blocking `facebook.com/tr` | Verify pixel in theme.liquid (just before `</head>`). Check that the live theme is the one you edited (Shopify → Themes → … → "Publish"). Check CSP headers (Cloudflare → Security → Events). |
| **D** (EC quality tier) | `quality_tier 'Unavailable'` | `Include user data` not enabled in the Google Ads conversion tag; Klaviyo-to-Google sync sending plaintext email instead of hashed | Tag Configuration → Advanced → Include user-provided data = ON. Verify Klaviyo Google Ads integration uses email-hashing (not plaintext). |
| **D** (EC email coverage) | `Email coverage 65.0% < 80%` | Guest checkouts not collecting email; Klaviyo profiles missing email field | Enable "Email collection on guest checkout" in Shopify. Backfill email field on Klaviyo profiles that have only phone or only address. |
| **E** (GA4 ↔ TW revenue) | `Revenue delta 22.0% > 5%` | TW over-counting (refund miscategorization) OR GA4 under-counting (expected post-iOS14.5) | If GA4 < TW by 5–30%: expected (document the delta as the "iOS14.5 baseline"). If GA4 < TW by >30%: investigate refund/return miscategorization in TW. If TW < GA4: a refund is being miscategorized or a test order leaked in. |
| **E** (order count delta) | `Order delta 8.0% > 3%` | Fulfillment webhook is delayed OR a payment failure is dropping the order | Check Shopify → Settings → Notifications → Order fulfillment. Check payment gateway → declined transactions report. |
| **F** (cohort roundtrip) | `Cohort match 70.0% < 95%` | Cohort names don't match exactly (case-sensitive) between Klaviyo and TW; Klaviyo → TW sync is firing on a different webhook | Verify cohort name in Klaviyo = TW (case-sensitive). Toggle Klaviyo integration in TW → Settings → Integrations. Verify sync is on `order.created` webhook (not `abandoned_cart`). |
| **G** (attribution drift) | `Match drift 12.0pp > 5pp target` | Something changed in the attribution stack since last week (new ad platform added; refund policy change; webhook delayed; pixel removed) | Check the diff in the last 7d's setup: any new ad platforms? Theme updates? Webhook changes? Policy changes? |

**Verification gate for Step 4:** after applying the fix, re-run the audit and confirm the previously-failing gate now passes. If it still fails, escalate to a Triple Whale support ticket (the most common unresolved case is a TW ↔ Meta CAPI integration that requires TW engineering to fix).

### Step 5 — Document the weekly audit cadence (one-time, ~15 min)

1. Create a Notion doc (or wherever the team tracks ops processes) titled "Move #6.5 Weekly Attribution Audit" with:
   - The 6 fixture files (linked to the canonical locations in `attribution_fixtures/`)
   - The cron schedule (Step 2's Monday 07:00 UTC line)
   - The on-call rotation (who gets paged when the audit fails)
   - The remediation table from Step 4 (or link to this playbook's Verification section)
2. Add a recurring calendar event: "Move #6.5 weekly audit review" (15 min, after the cron fires).
3. Add a Slack channel (or a #attribution-alerts channel if you have one) that receives the JSON output. Pin the message: "When the audit fails: open the JSON, look at the `remediation` field for the failed gate, follow the linked playbook section."

**Verification gate for Step 5:** the on-call rotation is documented AND the Slack channel is receiving the JSON output on the next scheduled run.

### Step 6 — Quarterly re-baseline (recurring, ~1 hr per quarter)

1. Every 90 days, run `python3 scripts/attribution_quality_audit.py --bootstrap ./attribution_fixtures/`. This regenerates the fixture files with canonical values; review each one and re-validate against the latest diagnostic exports.
2. Compare the quarterly re-baseline numbers against the previous quarter's numbers. If the new "canonical" values (the bootstrap's placeholder values) are far from your actual diagnostic numbers, the placeholder values are stale and the operator has been making decisions on bad data — escalate immediately.
3. Update the `attribution_drift.json` "previous" values to the current "current" values; the next audit compares against the new baseline.

**Verification gate for Step 6:** the new quarter's bootstrap fixtures match the actual diagnostic numbers within ±10% on all 4 numeric gates (A, C, D, E). If they don't, the audit has been silently reporting false-green for the past quarter.

## Common pitfalls

12 pitfalls with corrective "Fix:" lines.

1. **Running the audit before Move #6 is fully shipped.** Fix: re-run `python3 scripts/triple_whale_attribution_check.py` and confirm `overall_passed: True` before bootstrapping Move #6.5 fixtures. The Move #6.5 audit will report "Fixture file missing" for every gate if Move #6 isn't done — but more importantly, the audit is meaningless if the install is incomplete.
2. **Loosening thresholds to make gates pass.** Fix: NEVER loosen `MIN_META_CAPI_MATCH_RATE_PCT` below 90.0, `MAX_GA4_TW_REVENUE_DELTA_PCT` above 5.0, or `MIN_COHORT_ROUNDTRIP_MATCH_PCT` below 95.0 without a documented reason AND a `TICK: <date>` marker. The thresholds are pinned by `test_canonical_thresholds_published`; if you change them, also update the test (and the next operator will see a diff that explains why).
3. **Using stale fixture values.** Fix: re-export the diagnostic values every week. Stale values mean the audit is reporting the state of the attribution stack N weeks ago, not today. The bootstrap templates include placeholders; replace with real values before each weekly run.
4. **Ignoring Gate A when the match rate is 80–89%.** Fix: 80% is the throttling floor; below 90% you're losing campaign optimization. Even a "passing" 85% match rate is a yellow flag — investigate before the next campaign decision.
5. **Trusting the audit's PASS without checking the numbers.** Fix: even when the script reports `Summary: 6/6 gates passed`, eyeball the actual values. A 90.5% match rate is "passing" but the trend may be 95% → 92% → 90.5% (declining); the audit catches the floor, not the trend.
6. **Forgetting to update Gate G's "previous" values.** Fix: Gate G is a week-over-week comparison; without the previous value, the gate will fail with `drift_threshold_pct > 0` (because the diff is computed against the previous, not the threshold). Update the "previous" values weekly.
7. **Not setting up the cron / on-call rotation.** Fix: a weekly audit that nobody reads is a waste of time. The cron schedule (Step 2) + Slack channel (Step 5) + on-call rotation (Step 5) are the operational part of this move. Skipping them = audit results go to `/dev/null`.
8. **Treating the audit as a one-time task.** Fix: the audit's value compounds over time (Gate G is meaningless on the first run, useful on the second, critical on the tenth). Schedule the weekly cadence and don't skip weeks.
9. **Using Polar's diagnostics where the playbook assumes Triple Whale.** Fix: Polar Analytics has a different diagnostic-export UI (Polar → Settings → Diagnostics instead of Triple Whale → Settings → Diagnostics). The JSON fixture shape is the same (the audit script doesn't care which platform generated the data); just adjust the export path.
10. **Failing to act on a FAIL.** Fix: the audit's `remediation` field is the entry point; the Step 4 table is the action. If a FAIL is not actionable, escalate to Triple Whale / Polar / Meta / Google support within 7 days. A FAIL that's ignored for 30+ days is the same as not running the audit.
11. **Exporting the wrong time window.** Fix: ALL exports use "Last 7 days" as the canonical window. Different windows (last 14d, last 30d) produce different numbers and break the Gate G week-over-week comparison. Use exactly "Last 7 days" on every platform.
12. **Comparing Klaviyo cohort tags with TW cohort names without normalizing case.** Fix: the audit's Gate F is case-insensitive (so "Welcome Series" matches "welcome series"), but the cohort tag export from Klaviyo and the cohort name from TW may have different case conventions. If a cohort is "Welcome Series" in Klaviyo and "WELCOME SERIES" in TW, the audit's `lower()` comparison passes, but the human-readable cohort name in your dashboard will look like a mismatch. Pick one canonical casing and stick to it (recommend: Title Case).

## Verification

6 gates (A + C + D + E + F + G) that the attribution stack is producing trustworthy signal. Each gate is a numeric-threshold check, not just a key-presence check (Move #6's gates validate installation; Move #6.5's gates validate measurement quality).

- **Gate A — Meta CAPI event_id match rate + dedup ratio + coverage.** `python3 scripts/attribution_quality_audit.py --check meta_capi_match_rate` exits 0 if match rate ≥ 90% AND dedup ratio in [0.8, 1.5] AND coverage ≥ 95% of expected orders. The match rate is the killer metric — below 80% Meta throttles CAPI delivery; below 70% Meta drops the events entirely. **How to verify the gate is testing the right thing:** temporarily set `matched_events` to 100 (vs 950 expected = 10.5% match rate) and re-run — the gate should FAIL with `Match rate 10.5% < 90.0% target` in the detail.
- **Gate C — Meta Pixel coverage.** `--check meta_pixel_coverage` exits 0 if pixel coverage ≥ 95% of expected pageviews. **How to verify:** set `pixel_fired_count` to 5000 (vs 10000 expected = 50% coverage) and re-run — should FAIL with `Pixel coverage 50.0% < 95.0% target`.
- **Gate D — Google Enhanced Conversions quality tier + email coverage.** `--check google_enhanced_conversions_quality` exits 0 if quality_tier ∈ {Good, Excellent} AND hashed_email_coverage_pct ≥ 80%. **How to verify:** set `quality_tier` to "Unavailable" — should FAIL with `quality tier 'Unavailable' not in ['Excellent', 'Good']`.
- **Gate E — GA4 ↔ Triple Whale revenue delta + order count delta.** `--check ga4_tw_revenue_delta` exits 0 if `|GA4 - TW| / TW ≤ 5%` AND `|actual - expected| / expected ≤ 3%`. **How to verify:** set `ga4_revenue_last_7d` to 30000 (vs tw=67500 = 55.6% delta) — should FAIL with `revenue delta 55.6% > 5.0% target` AND `iOS14.5` in the remediation text.
- **Gate F — Klaviyo ↔ Triple Whale cohort roundtrip.** `--check klaviyo_tw_cohort_roundtrip` exits 0 if ≥ 95% of sample orders have matching cohort assignments (case-insensitive) across Klaviyo and Triple Whale. **How to verify:** change 1 of 5 sample orders' `tw_cohort` to "Different Cohort" — should FAIL with `80.0% < 95.0% target`.
- **Gate G — Attribution drift week-over-week.** `--check attribution_drift` exits 0 if abs(current - previous) ≤ 5pp on both match rate and revenue delta. **How to verify:** set `current_match_rate_pct` to 80 (vs previous 95 = 15pp drift) — should FAIL with `match rate drift 15.0pp > 5.0pp target`.

**To run the full audit:** `python3 scripts/attribution_quality_audit.py` (human-readable) or `python3 scripts/attribution_quality_audit.py --json` (machine-readable, suitable for cron). Exit code 0 = all gates pass; 1 = at least one gate fails.
