## Goal

By the end of this playbook, an operator with Move #6 + #6.5 + #6.6 + #6.7 + #6.8 all live can:

1. **Run one command** that consumes the Move #6.8 cross-platform rollup JSON, applies the canonical 5-decision-rule engine, and either POSTs one Slack-compatible alert payload OR writes one timestamped archive file to `--alert-archive` (the hermetic fallback when `--webhook-url` is omitted).
2. **Receive one alert per cycle** with the canonical 13-field alert-payload shape [alert_id + timestamp + source + severity + title + summary + per_platform_breakdown + drift_summary + root_cause_hypothesis + remediation + overall_passed + thresholds_used + raw_rollup_path] — downstream consumers can parse without surprise, the payload shape is pinned by `_canonical_alert_shape_published()` regression gate.
3. **Surface cross-platform drift + root-cause hypothesis + remediation inline** — the alert payload pulls Move #6.8's `root_cause_hypothesis.id + label + remediation` fields directly into the Slack message body so the operator reads the suspected shared root cause + the corrective action WITHOUT having to open the rollup JSON.
4. **Respect a 3600-second cooldown** — if a recent alert (within `cooldown_seconds`) exists in the `.alerts/` directory, the script SUPPRESSES the duplicate POST (cooldown is overridable via `--cooldown-seconds <N>` or `--skip-cooldown` for force-fire).
5. **Fall back to local archive hermetically** — when `--webhook-url` is omitted, the script writes `alert-<ISO-timestamp>-<severity>.json` to `--alert-archive` (default `./.alerts/`) so operators can ship the webhook without exposing any external endpoint.

## Step-by-step

### Step 1 — Install the webhook script (5 min)

```bash
# Sanity check the companion script is present
cd /data/workspace/ecommerce-ops
test -f scripts/attribution_health_alert_webhook.py && echo "OK: webhook script present"
test -f scripts/tests/test_attribution_health_alert_webhook.py && echo "OK: test file present"

# Run the full TDD test suite to verify the canonical regression gates are GREEN
python3 scripts/tests/test_attribution_health_alert_webhook.py main_test 2>&1 | tail -5
# Expected: 38/38 PASS zero failures (ThresholdConstantsPinned + AlertShapePinned + IsHttpUrl + ...)

# Verify the Move #6.8 rollup script is also present (Move #6.10 consumes its output)
test -f scripts/attribution_cross_platform_rollup.py && echo "OK: rollup script present"
```

**Decision gate before Step 2:** if the webhook script is missing OR the test suite is RED, STOP. Fix the underlying script (re-run `python3 scripts/tests/test_attribution_health_alert_webhook.py main_test` and read the failing test class) BEFORE proceeding with the cron wiring. The webhook does NOT paper over broken Move #6.8 substrate.

### Step 2 — Bootstrap the archive directory (2 min)

```bash
# Bootstrap creates the .alerts/ archive directory the script writes to when --webhook-url is omitted
python3 scripts/attribution_health_alert_webhook.py --bootstrap /data/workspace/ecommerce-ops
# Expected: "Bootstrap: created /data/workspace/ecommerce-ops/.alerts/" on stderr + exit code 0

# Verify the directory exists and is writable
test -d /data/workspace/ecommerce-ops/.alerts && echo "OK: archive directory ready"
```

**Why a separate archive directory matters.** Even when you wire a Slack webhook, the archive directory is the canonical fallback for: (a) `--skip-webhook` debugging runs, (b) `--validate-thresholds` regression-check runs, (c) post-mortem analysis when a webhook POST fails (the archive has the exact payload that was attempted), (d) the cooldown walker (`_respect_cooldown` reads the most recent archive file timestamp). Without `--bootstrap`, the first failure cycle will silently skip the archive write.

### Step 3 — Validate the canonical thresholds + alert shape (2 min)

```bash
# Regression gate: verify the 5 canonical webhook thresholds + 13 canonical alert-payload fields are STILL published
python3 scripts/attribution_health_alert_webhook.py --validate-thresholds
# Expected: prints the canonical thresholds dict + the canonical 13-field alert-payload list, exit code 0
```

**This gate is the canonical Move #6.10 regression-check pattern** (mirrors Move #6.8's `--validate-thresholds`). If you ever see the canonical 5 thresholds or the 13-field alert-payload shape CHANGE without a documented `TICK: <date>` marker in `docs/journal.md`, treat it as a regression — the regression-gate tests `ThresholdConstantsPinned` + `AlertShapePinned` will fail in the test suite, AND the script will fail this `--validate-thresholds` CLI call. **The thresholds + alert shape are pinned because downstream consumers parse on them — silently renaming a field or lowering a threshold breaks every Slack message + every archive file + every cooldown walker that depends on the shape.**

### Step 4 — Wire the cron (5 min)

```bash
# Add the webhook to crontab (replace /data/workspace/ecommerce-ops with your path)
# Runs every Monday at 09:05 local time, AFTER Move #6.8 (which runs at 09:00 per playbook 06.8)
5 9 * * 1 cd /data/workspace/ecommerce-ops && python3 scripts/attribution_health_alert_webhook.py \
  --rollup-json /var/log/attribution-rollup-latest.json \
  --webhook-url "$ATTRIBUTION_ALERT_WEBHOOK_URL" \
  --alert-archive /var/log/attribution-alerts/ \
  --cooldown-seconds 3600 \
  >> /var/log/attribution-webhook.log 2>&1
```

**Cron timing matters.** The webhook MUST run AFTER Move #6.8 (so the rollup JSON is fresh) and BEFORE any human is likely to look at the dashboards (so the Slack alert is in the channel by 09:30 Monday). The default 09:05 Monday (5 minutes after Move #6.8's 09:00) is canonical. **Never run Move #6.10 BEFORE Move #6.8** — the webhook will read a stale or missing rollup JSON and emit a "ROLLOUP FILE MISSING" sentinel alert.

### Step 5 — Configure Slack alert format (5 min)

The webhook emits **one Slack message per cycle** with this structure (rendered via Slack's native message-formatting when the JSON payload contains a `text` block — see Pitfall #4 for the fallback path):

```
🔴 ATTRIBUTION HEALTH ALERT: 2 per-platform audit(s) failed gates: Meta, Snap
*Per-platform status:* 2/5 platforms failing
  - Meta CAPI match rate: 87.2% (fail, threshold ≥90%) — likely cause: theme.liquid snippet update dropped fbq('init')
  - Snap EMQ: 65.4% (fail, threshold ≥70%) — likely cause: Snap Advanced Matching toggle flipped off
*Cross-platform drift:* 2 platforms with simultaneous match-rate drops (Meta -3.4pp + Snap -4.1pp)
*Root-cause hypothesis:* iOS consent banner change on 2026-MM-DD dropped hashed-email match rate on both
*Remediation:* revert consent banner to pre-2026-MM-DD version; verify fbq('init') is still in theme.liquid; re-enable Snap Advanced Matching toggle
*Thresholds used:* match-rate-drift 3.0pp / coverage-drift 2.0pp / cooldown 3600s
```

On a healthy cycle (no fires):

```
[OK] No alert fired (all gates passing, no drift detected)
```

On a cross-platform-drift-only fire (no per-platform fails, severity="warning"):

```
⚠️ ATTRIBUTION DRIFT WARNING: cross-platform drift detected: match-rate drift 3.2pp / coverage drift 1.1pp / 2 platforms with match-rate drop
*Per-platform status:* 5/5 platforms passing (Meta CAPI 91.4% / GA4↔TW delta 1.5% / Google EC Good / TikTok EAPI 87.8% / Snap CAPI 84.2% / Pinterest CAPI 89.1% / Snap EMQ 74.8% / Pinterest EM 78.3%)
*Cross-platform drift:* 2 platforms with simultaneous match-rate drops (Meta -3.2pp + TikTok -3.1pp)
*Root-cause hypothesis:* theme.liquid update on 2026-MM-DD dropped pixel coverage on Meta + TikTok
*Remediation:* verify the latest theme.liquid deploy did not remove the Meta + TikTok pixel snippets; re-test pixel coverage via Move #6.5 + #6.6
```

### Step 6 — Investigate any failures (15-30 min per cycle)

When the webhook Slack alert fires (severity="critical" or "warning"):

1. **Read the `root_cause_hypothesis` field FIRST.** This is Move #6.8's canonical 5-hypothesis matcher (theme.liquid update / CAPI token rotation / iOS consent banner change / app uninstall / no shared hypothesis). If the hypothesis points at a shared root cause, fix that FIRST.
2. **Read the `remediation` field.** This is the corrective action Move #6.8 matched for the hypothesis. Apply the remediation, then re-run Move #6.5/6.6/6.7 individually to confirm the gates recover.
3. **Inspect the `per_platform_breakdown` array.** This enumerates each platform's `gates_failing` array (sorted alphabetically) + `exit_code` + `error`. Open the matching playbook (06.5 / 06.6 / 06.7) for the specific remediation steps.
4. **If `cooldown_status.applied == true`**, the alert was SUPPRESSED (a fresh alert existed within the cooldown window). Read `cooldown_status.reason` to see the elapsed-seconds; this is informational, not an error.
5. **Inspect `--alert-archive/<alert-id>.json`** for the exact payload + `post_result` (HTTP status code, error message if any). This is the canonical post-mortem source.

| **Mean time to resolution (MTTR)** for attribution regressions | Slack-thread close timestamp + archive file timestamp | <48h from Slack alert to fix | Identify the gap (most often: theme.liquid snippet update without re-testing per Move #6.5/6.6/6.7 Pitfall #2) |

## Common pitfalls

- **Pitfall #1: Running Move #6.10 BEFORE Move #6.5/6.6/6.7/6.8 are all shipped.** If any per-platform audit has failing gates, Move #6.8 inherits the failures and Move #6.10 emits a misleading "cross-platform drift" alert with the wrong root-cause hypothesis. **Fix:** run each per-platform audit individually FIRST, confirm `overall_passed: true` for all 3, run Move #6.8 end-to-end, THEN install Move #6.10. The webhook is a notification dispatcher, not a fix for broken Move #6.8 substrate.
- **Pitfall #2: Loosening the canonical webhook thresholds (5-decision-rule engine constants) to make alerts stop firing.** The canonical defaults [Rule 1: ANY per-platform fail / Rule 2: cross-platform drift detected / Rule 3: match-rate drift > 3.0pp / Rule 4: coverage drift > 2.0pp / Rule 5: cooldown 3600s] are tuned to catch ~95% of real regressions without spamming. Loosening them silences real alerts. **Fix:** NEVER loosen the canonical webhook thresholds without a documented reason AND a `TICK: <date>` marker in `docs/journal.md`. The thresholds are pinned by `test_canonical_thresholds_published` in the companion test suite + by the `--validate-thresholds` regression-gate CLI mode.
- **Pitfall #3: Renaming or adding fields to the 13-field canonical alert-payload shape without updating the regression-gate test.** The shape is pinned by `_canonical_alert_shape_published()`; downstream consumers (Slack message-formatting, archive-file parsers, cooldown walker) depend on the canonical 13-field order. Adding fields without updating the regression gate silently breaks the contract. **Fix:** if you need to add a field, update `ALERT_PAYLOAD_TOP_LEVEL_FIELDS` list + add a `TICK: <date>` marker + update the test class `AlertShapePinned` to assert the new field. Renaming is forbidden — instead, add a new field and deprecate the old.
- **Pitfall #4: Sending the webhook payload as `application/x-www-form-urlencoded` (legacy Slack format) instead of `application/json`.** The Move #6.10 webhook uses the modern Slack Incoming-Webhook shape (`Content-Type: application/json` with the alert payload as the JSON body). The legacy shape (`payload=<urlencoded-json>`) is silently rejected by newer Slack workspaces. **Fix:** verify the `_post_webhook` function uses `headers={"Content-Type": "application/json"}` (it does in the canonical implementation); if you fork the script, preserve the JSON Content-Type header.
- **Pitfall #5: Treating Move #6.10 as a REPLACEMENT for Move #6.8.** Move #6.10 is a NOTIFICATION DISPATCHER for Move #6.8's output. If you stop running Move #6.8 and only run Move #6.10, the webhook reads a stale rollup JSON (or no JSON at all) and emits the sentinel "ROLLOUP FILE MISSING" alert every cycle. **Fix:** Move #6.10 RUNS AFTER Move #6.8; both cron entries must remain. The canonical timing is Move #6.8 at 09:00 + Move #6.10 at 09:05 (5-minute gap).
- **Pitfall #6: Setting the webhook URL in the cron line instead of in an env var.** If the webhook is in the crontab file in plaintext, anyone with read access to the crontab can post to your Slack channel. **Fix:** put the webhook in `/etc/attribution-webhook.env` (mode 600, owned by the cron user) and source it in the cron entry: `5 9 * * 1 source /etc/attribution-webhook.env && cd /data/workspace/ecommerce-ops && python3 scripts/attribution_health_alert_webhook.py ...`
- **Pitfall #7: Forgetting to bootstrap the `.alerts/` archive directory.** Without `--bootstrap`, the first failure cycle silently skips the archive write AND the cooldown walker (`_respect_cooldown`) cannot find any archive files to compare against. **Fix:** run `--bootstrap /data/workspace/ecommerce-ops` once during Step 2 of this playbook; the script creates `./.alerts/` under the given path. Verify with `ls -la /data/workspace/ecommerce-ops/.alerts/` before the first cron run.
- **Pitfall #8: Running the webhook on a different host from Move #6.8.** If Move #6.10 runs on a different host, it needs to fetch the Move #6.8 `--output-json` over the network (slow + auth required) OR read a stale local copy. **Fix:** run Move #6.10 on the SAME host as Move #6.8 so it can read the local `--output-json` file directly. The default cron entry assumes this.
- **Pitfall #9: Assuming Move #6.10 fixes the underlying attribution problems.** Move #6.10 is a NOTIFICATION DISPATCHER. It detects drift + dispatches the alert; it does NOT fix the underlying problems. The actual fixes are in Move #6.5/6.6/6.7 (rebuild pixel snippets, refresh CAPI tokens, re-enable Advanced Matching). **Fix:** when the webhook fires, route to the per-platform playbook (06.5 / 06.6 / 06.7) for the actual remediation steps. The `remediation` field in the alert payload surfaces Move #6.8's matched root-cause corrective action; follow it, then re-run Move #6.5/6.6/6.7 individually.
- **Pitfall #10: Setting `--cooldown-seconds` too high (e.g. 86400 = 24 hours) thinking it prevents alert fatigue.** A 24-hour cooldown means a single regression fires at most once per day — but if the regression persists for multiple days, only the FIRST day's alert fires. The operator misses the Day-2 / Day-3 escalations. **Fix:** keep `--cooldown-seconds` at the canonical 3600 (1 hour) for weekly Path B cadence. If you genuinely need daily-only alerting, use a cron filter (`0 9 * * 1` runs weekly at 09:00 Monday, NOT every day) — never extend cooldown to suppress legitimate multi-day alerts.
- **Pitfall #11: Setting `--cooldown-seconds` too low (e.g. 0 = no cooldown).** Without cooldown, the webhook fires EVERY cycle the drift is detected — for Move #6.8 on Path C daily cadence, that's 7 alerts per week per regression. Slack `#attribution-alerts` channel drowns in duplicates. **Fix:** keep `--cooldown-seconds` at the canonical 3600 minimum. If you need to allow back-to-back alerts (e.g. for testing), use `--skip-cooldown` per-run, NOT a lowered cooldown.
- **Pitfall #12: Treating the archive directory as a transient debug artifact.** The `.alerts/` directory is the canonical MOVE #6.10 PERSISTENT STATE — the cooldown walker reads the most recent archive file's timestamp on every cycle. If you delete the archive directory between cycles, the cooldown resets and the next drift cycle fires immediately. **Fix:** NEVER `rm -rf .alerts/` between cycles. If you need to reset cooldown for testing, use `--skip-cooldown` per-run; if you need to clear old alerts, archive them to S3/GCS cold storage (NOT delete).
- **Pitfall #13: Forgetting the canonical 5-minute gap between Move #6.8 (09:00) and Move #6.10 (09:05).** If both run at 09:00 in parallel, the webhook may read a partial / in-progress rollup JSON (Move #6.8 has not finished writing yet). The `_load_rollup_json` sentinel fallback will fire and emit a misleading "ROLLOUP FILE MISSING" alert. **Fix:** always run Move #6.10 AT LEAST 1 minute after Move #6.8 (the canonical 5-minute gap provides a generous safety margin). For Path C daily cadence (Move #6.8 daily), use a 5-minute gap; for hourly cadence, use a 1-minute gap.
- **Pitfall #14: Wiring the webhook without `--validate-thresholds` daily cron.** Without the daily regression-check, a silent contributor change to the canonical 5 thresholds or 13-field alert-payload shape will go undetected until a downstream consumer breaks (e.g. Slack message-formatting fails, archive parser crashes). **Fix:** wire `--validate-thresholds` to a daily cron at 09:00 — exit-code-only, no Slack post. Alert on non-zero exit code via the canonical Slack-channel-health-monitor pattern.
- **Pitfall #15: Assuming `--skip-webhook` means "don't archive either".** `--skip-webhook` suppresses the POST to the webhook URL but STILL writes the archive file + prints the payload to stdout/stderr. This is intentional for hermetic-mode debugging. **Fix:** when iterating locally with `--skip-webhook`, verify the archive file is written (`ls -la .alerts/` after each run) — if it's missing, the script crashed before reaching the archive step (usually a JSON parsing error in `_load_rollup_json`).

## Companion tool

`scripts/attribution_health_alert_webhook.py` (~555 lines, Archetype C/D-light hybrid — consumes Move #6.8 cross-platform rollup JSON, applies the canonical 5-decision-rule engine, POSTs to a Slack-compatible webhook URL via stdlib `urllib.request` OR falls back to a local archive directory when `--webhook-url` is omitted; 13-field canonical alert-payload shape pinned by `_canonical_alert_shape_published()` regression gate; 5 canonical webhook thresholds pinned by `_canonical_thresholds_published()` regression gate; 3 exit codes [0 = no alert / 1 = alert fired + posted OR archived / 2 = alert fired but webhook POST FAILED — operator MUST investigate]; 3-mode CLI [default mode: consume rollup + dispatch / `--bootstrap` mode: create `.alerts/` directory / `--validate-thresholds` mode: regression gate verifying the canonical thresholds + alert shape are STILL published]). **Hermetic — stdlib `urllib.request` only; no `requests` dependency, no `slack_sdk` dependency, no paid SaaS integration. The webhook URL is the only external resource; when omitted, the script is fully local.**

**CLI:**

```bash
# Run end-to-end (consume Move #6.8 rollup + dispatch to Slack + archive)

  --rollup-json /var/log/attribution-rollup-latest.json \
  --webhook-url "$ATTRIBUTION_ALERT_WEBHOOK_URL" \
  --alert-archive /var/log/attribution-alerts/ \

# Bootstrap the archive directory (one-time setup)

# Run end-to-end + JSON output (for monitoring / debugging)
python3 scripts/attribution_health_alert_webhook.py \

# Validate the canonical thresholds + alert shape are STILL published (regression check)

# Skip the webhook POST but still build + archive + print the payload (hermetic debugging)

# Force-fire even if a recent alert exists (testing only — never use in production cron)
```
